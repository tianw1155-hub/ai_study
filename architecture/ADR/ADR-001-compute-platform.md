# ADR-001: MVP 部署架构选择

**状态**: 已接受  
**日期**: 2026-03-22  
**决策者**: 架构团队

---

## 背景/问题陈述

MVP 需要在一周内交付第一个可用原型。团队 10 人以内，无专职运维。需要选择一种既能快速交付、又能在验证 PMF 后容易扩展的部署架构。

核心问题：
1. 如何最小化运维负担？
2. 如何实现按需扩缩容？
3. 如何控制初期成本？

---

## 候选方案

### 方案 A: SCF 函数计算

| 维度 | 评估 |
|------|------|
| 运维复杂度 | **极低** - 免运维，云厂商托管 |
| 扩缩能力 | **自动** - 按请求数自动扩缩 |
| 冷启动 | 1-3 秒 (可通过预置并发优化) |
| 成本 | **按调用计费** - 无流量时近乎零成本 |
| 供应商锁定 | 中等 - 阿里云 FC / AWS Lambda 有类似 API |

### 方案 B: CVM 轻量应用服务器

| 维度 | 评估 |
|------|------|
| 运维复杂度 | **高** - 需自行管理服务器 |
| 扩缩能力 | 需手动或配置 ASG |
| 冷启动 | 无 |
| 成本 | 固定月费，即使无流量也计费 |
| 供应商锁定 | 低 - 标准 Linux 环境 |

### 方案 C: TKE 容器编排

| 维度 | 评估 |
|------|------|
| 运维复杂度 | **高** - K8s 学习曲线陡 |
| 扩缩能力 | 自动 |
| 冷启动 | 10-30 秒 |
| 成本 | 相对较高 |
| 供应商锁定 | 高 - TKE 特定配置 |

---

## 决策

**选择方案 A: SCF 函数计算**

### 决策理由

1. **团队现状匹配**: 10 人团队，无运维，SCF 的免运维特性可将运维工作量降至 0
2. **MVP 时间线**: 一周交付需要最大化开发效率，SCF 的快速部署能力是关键
3. **成本模型**: MVP 初期流量未知，按调用计费可避免固定成本浪费
4. **技术债务可控**: SCF 迁移到 CVM/TKE 的成本可接受

### 限制条件

- 冷启动延迟需通过预置并发优化
- 有状态场景使用 Redis/MongoDB 外部化
- 避免大单体函数，保持函数粒度合理

---

## 后果

### 正面

- 运维工作量接近零
- 成本与实际流量成正比
- 自动扩缩容，无需人工干预
- 快速迭代，专注业务逻辑

### 负面

- 冷启动延迟需额外优化
- 有状态处理需要外部服务
- 函数粒度设计需要经验
- 厂商锁定中等
