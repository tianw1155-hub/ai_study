### 1.2 新发现异常场景（E-10 ~ E-14）

> 以下场景在当前 PRD 中**未被覆盖**，建议补充：

| # | 异常场景 | 风险等级 | 建议处理方式 |
|---|----------|----------|--------------|
| **E-10** | **GitHub API 限流（403 Rate Limit）** | 高 | Step 2 Git revert 前先查询 rate limit 剩余量；限流时 sleep + 重试，记录到 rollback_logs step=2 |
| **E-11** | **Step 1 数据库更新成功，Step 2 Git revert 超时但实际成功**（网络抖动导致请求超时，但 GitHub 端已创建 revert commit） | 高 | Step 2 需要**幂等性设计**：revert 前先查询目标 commit 是否已是最新 commit；或使用 GitHub API 的 idempotency key |
| **E-12** | **Step 3 部署触发后，Vercel/Render API 返回 500** | 中 | rollback_logs step=3 标记 failed；前端提供"重新部署"按钮；Step 1-2 状态保留 |
| **E-13** | **Step 3 部署触发后，实际部署失败（Vercel 返回 failed 状态）** | 中 | 前端展示具体失败原因（枚举：网络错误/权限不足/构建失败/超时）；更新 rollback_logs step=3 status=failed |
| **E-14** | **Step 1 数据库操作失败（写入 rollback_logs 或更新 version 失败）** | 高 | 直接返回错误，不继续；事务回滚；用户看到错误提示 |

### 1.3 E-1 细化问题：Git revert 失败后的"手动介入"

**当前描述：**
> "Git revert 失败需人工处理"

**问题：**
1. "人工处理"由谁执行？用户？管理员？
2. 人工处理完成后，系统如何识别并恢复断点续做？
3. 人工处理后，rollback_logs 的状态如何流转？

**建议补充：**
- 管理员收到告警后，在 GitHub 手动解决冲突并 commit
- 系统提供"标记为已手动处理"按钮，管理员点击后将 rollback_logs step=2 标记为 completed，允许用户继续 Step 3

## 二、rollback_logs 表设计评审

### 2.1 缺失字段

| 字段 | 必要性 | 原因 |
|------|--------|------|
| `retry_count` | **必须** | 当前只记录 step 状态，未记录每个 step 的重试次数。E-1 场景下"最多重试 3 次"无法落地验证 |
| `github_ref` | **建议** | Step 2 回退后新建的 revert commit SHA 应记录，用于后续排查 |
| `deployment_id` | **建议** | Step 3 触发的 Vercel/Render deployment ID 应记录，用于关联部署状态查询 |

**建议新增字段：**
```sql
ALTER TABLE rollback_logs ADD COLUMN retry_count INT DEFAULT 0;  -- 每个 step 的重试次数
ALTER TABLE rollback_logs ADD COLUMN github_revert_sha VARCHAR;  -- revert commit SHA
ALTER TABLE rollback_logs ADD COLUMN deployment_id VARCHAR;       -- 部署任务 ID
```

### 2.2 断点续做机制缺失

当前描述了"继续回退"的概念，但以下问题未解答：

1. **"继续回退"从哪里恢复？** 从当前 step 的 failed 状态继续，还是从头重跑？
2. **Step 2 Git revert 幂等性未定义**：如果 Step 2 failed，用户点击"继续"，重新调用 Git revert API 是否会产生重复 commit？
3. **Step 1-2 完成，Step 3 failed，"继续"时是否重新触发部署？** 还是复用已 revert 的 commit？

**建议：** 断点续做时，以 rollback_logs 中 `step` 字段为准，从 `status=pending` 或 `status=failed` 的 step 继续，并确保每个 step 的操作是幂等的。

## 三、验收标准可测试性评审

### 3.1 Must 类（共 6 项）

| ID | 标准 | 可测试性 | 测试难点 |
|----|------|----------|----------|
| M-1 | GFM Markdown 渲染正确率 = 100% | ✅ 可测试 | 需要构造所有 GFM 语法节点（表格、代码块、任务列表、脚注等）的测试用例，与标准渲染器比对节点数 |
| M-2 | GitHub 链接 2s 内跳转 | ✅ 可测试 | 直接计时 |
| M-3 | 代码目录树 100% 匹配 | ✅ 可测试 | 对比 GitHub API tree 接口与前端展示 |
| M-4 | 预览链接 HTTP 200 在 30s 内 | ✅ 可测试 | 自动化轮询 + 计时 |
| M-5 | 回退后三端一致，< 3min | ⚠️ 部分可测试 | "三端一致"需要三方数据比对，可测试；但"更新 < 3min"是性能指标，需明确定义从哪个时间点开始计时 |
| M-6 | 部署失败原因枚举明确 | ✅ 可测试 | Mock Vercel/Render 返回各种错误场景，验证前端提示文案 |

### 3.2 Should 类（共 4 项）

| ID | 标准 | 可测试性 | 测试难点 |
|----|------|----------|----------|
| S-1 | Diff 视图 100% 展示，无遗漏 | ✅ 可测试 | 需准备 ground truth diff 样例（含边界：空diff、单行diff、大文件diff），逐一验证 |
| S-2 | 日志延迟 < 2s | ✅ 可测试 | 端到端时间戳打点，注意网络波动影响 |
| S-3 | 单文件预览 < 500ms（< 100KB） | ✅ 可测试 | 自动化计时 |
| S-4 | 部署历史 localStorage 不丢失 | ⚠️ 需明确范围 | localStorage 属于前端存储，前端单测可覆盖；但跨浏览器/隐私模式场景未定义 |

### 3.3 指标歧义需澄清

| 问题项 | 当前描述 | 歧义点 | 建议澄清 |
|--------|----------|--------|----------|
| M-5 "更新时间 < 3min" | "回退后代码/PRD/预览三端一致，更新时间 < 3min" | "更新时间"指每个步骤耗时，还是整体回退完成时间？3min 从哪个节点开始计时？ | 应明确：整体回退完成时间 ≤ 3min；计时起点 = 用户点击"确认回退"，终点 = 预览链接指向回退版本 |
| M-1 "100% 渲染正确率" | "GFM Markdown（含表格、代码块、任务列表）渲染正确率 = 100%" | "正确率 100%"是目视无错，还是 AST 节点级比对？ | 建议：GFM 语法节点（heading、table、code_block、task_item、blockquote 等）渲染后 DOM 结构与 reference 实现一致，100% 匹配 |

## 四、新发现的问题

### 4.1 高优先级

| # | 问题 | 影响 |
|---|------|------|
| P1 | **Step 2 Git revert 无幂等性保障** | 网络超时场景下，retry 可能产生多个重复 revert commit，污染仓库历史 |
| P2 | **rollback_logs.retry_count 缺失** | "每个步骤最多重试 3 次"无法落地执行和验证 |
| P3 | **E-1 Git revert 失败后的"手动介入"流程未闭环** | 人工处理后系统无法识别，回退流程卡死 |
| P4 | **E-10 GitHub API 限流未覆盖** | 回退操作大量调用 GitHub API，生产环境可能触发限流导致失败 |

### 4.2 中优先级

| # | 问题 | 影响 |
|---|------|------|
| P5 | **E-8 "中止部署"选项的触发条件未明确** | 等待 vs 中止的决策逻辑未定义 |
| P6 | **S-4 localStorage 跨端同步未定义** | 同一用户多设备登录时，部署历史记录不一致 |
| P7 | **M1.2 版本数量上限 50 个，但归档触发时机描述为 "> 50"** | 边界值：第 50 个版本是否归档？建议改为 "≥ 50" 或 "50 个版本时，第 51 个触发归档" |
| P8 | **M4.4 回退通知"回退步骤 {n}/3"——但实际步骤是 4 步** | 文档存在数字不一致：通知说 {n}/3，流程图是 Step 1~4 |

### 4.3 低优先级

| # | 问题 | 建议 |
|---|------|------|
| L1 | M3.4 部署历史"无需触发新回退"——但实际更新预览链接的机制未说明 | 补充：预览链接切换是否需要重新部署，还是通过 Vercel/Render 的 alias 功能直接切换？ |
| L2 | C-1 PR 自动创建描述为"不自动创建 PR"——但"不自动"是设计决策，无需放在验收标准 Could 栏 | 建议移至 Won't 栏或改为"W-5" |

## 五、评审总结

### 5.1 整体评价

| 维度 | 评分 | 说明 |
|------|------|------|
| 需求完整性 | ⭐⭐⭐⭐ (4/5) | 核心场景覆盖较全，异常处理有体系 |
| 验收标准可测试性 | ⭐⭐⭐ (3/5) | 大部分可测试，但部分指标（M-1/M-5）存在歧义需澄清 |
| 异常场景完整性 | ⭐⭐⭐ (3/5) | 覆盖了主要 9 个场景，但 E-10~E-14 等新发现场景缺失 |
| 表设计完整性 | ⭐⭐⭐ (3/5) | 核心字段够用，但 retry_count 等关键字段缺失 |

### 5.2 必须修复项（建议 v0.3 纳入）

1. **P1**: Step 2 Git revert 加入幂等性设计（先查 commit 状态或使用 idempotency key）
2. **P2**: rollback_logs 表增加 `retry_count` 字段
3. **P4**: 补充 E-10 GitHub API 限流处理
4. **P8**: 回退通知"步骤 {n}/3"改为"步骤 {n}/4"（与流程图步骤数一致）

### 5.3 建议优化项（v0.3+）

1. E-1 手动介入流程闭环设计
2. P7 版本归档边界值澄清（50 vs >50）
3. M-5/M-1 验收标准歧义澄清
4. 断点续做机制细化（恢复起点 + 幂等性）
