## 一、PRD-首页.md 评审

### 🔴 严重问题

**I-1: WebSocket 认证方式与 API 签名文档不一致**
- **位置：** M2.3 WebSocket 实时推送 vs 第6章 API 接口签名
- **问题描述：** M2.3 明确说明 JWT Token 通过 `Sec-WebSocket-Protocol` 头传递（安全性修复），但第6章 API 接口签名中仍写 `wss://api.devpilot.com/ws?token={jwt}`
- **影响：** 测试用例无法明确认证方式，可能按错误方式编写自动化脚本
- **建议：** API 接口签名应与 M2.3 保持一致，改为 `wss://api.devpilot.com/ws`（认证通过协议头），并补充说明 JWT 通过 `Sec-WebSocket-Protocol` 头传递

### 🟡 中等问题

**I-2: S-2 敏感词检测测试集规模不明确**
- **位置：** Should 验收标准 S-2
- **问题描述：** 验收标准要求"正向≥100条，负向≥50条，含拼音/特殊字符边界"，但 PRD 未明确说明测试集的具体构造方式和标准，测试结果难以客观判定
- **影响：** 无法设计可重复的自动化测试
- **建议：** 补充测试集规格文档（如拼音变体、emoji 替代、特殊字符混淆等），或明确测试由人工+工具协同完成

**I-3: 缺少登录状态的完整流程测试用例覆盖**
- **位置：** US-1/US-2 与 5.5 登录状态说明
- **问题描述：** 验收标准 M-1/M-2 未覆盖"未登录状态提交需求→跳转登录→登录后继续提交"的完整流程场景
- **影响：** 核心用户路径未完整测试
- **建议：** 补充登录拦截→OAuth跳转→登录后状态恢复的测试场景

### 🟢 轻微问题

**I-4: S-1 自动跳转时间定义模糊**
- **位置：** Should 验收标准 S-1
- **问题描述：** "跳转时间 1.5 秒"未明确是"提交成功后 1.5 秒开始跳转"还是"跳转动画耗时 1.5 秒"
- **建议：** 明确计时起点（如"提交成功回调 → 开始跳转的延迟"）

## 二、PRD-任务看板.md 评审

### 🔴 严重问题

**I-5: 重试机制时间间隔无验收标准**
- **位置：** M2.5 重试机制定义
- **问题描述：** 重试间隔明确定义为指数退避（10s → 30s → 90s），但 **Should 验收标准中无任何对应测试指标**，M-6 仅验证 retryCount +1，未验证时间间隔
- **影响：** 重试间隔不符合预期时无法被测试发现
- **建议：** 补充 S-x 验收标准："重试间隔符合指数退避（允许 ±10% 误差）"，并明确测量方式（时间戳打点）

### 🟡 中等问题

**I-6: M-3 前端渲染延迟指标与系统约束矛盾**
- **位置：** Must 验收标准 M-3
- **问题描述：** M-3 定义"WebSocket 推送接收后渲染延迟 < 500ms"，但下方 M-3 指标澄清说明"SYSTEM PRD 规定 Go API 轮询 Temporal 间隔为 3s，后端状态同步延迟下限为 3s"。这导致前端无论如何优化，总延迟仍受限于 3s
- **影响：** 验收标准存在逻辑矛盾，可能导致测试误判
- **建议：** 明确区分"前端纯渲染延迟"和"端到端状态同步延迟"两个指标，避免混淆

**I-7: 取消任务测试覆盖状态不完整**
- **位置：** M3.3 操作按钮 vs M-5 验收标准
- **问题描述：** M-5 仅验证"取消后任务从看板移除（进入 cancelled）"，但 M3.3 表格显示取消按钮条件为 `pending / running / testing` 三种状态，未明确测试是否覆盖所有三种状态
- **影响：** 某些状态下取消功能可能失效未被测试发现
- **建议：** M-5 补充"取消操作需覆盖 pending、running、testing 三种状态"

**I-8: 虚拟滚动 S-1 帧率测试条件不明确**
- **位置：** Should 验收标准 S-1
- **问题描述：** "单列 > 50 个任务时滚动流畅（60fps）"未定义测试设备和测试方法（人工感受还是自动化帧率采集？）
- **影响：** 测试结果难以客观量化
- **建议：** 明确测试方法（如 Chrome DevTools Performance 面板、Lighthouse）或定义自动化帧率采集脚本

### 🟢 轻微问题

**I-9: 状态机流转图缺少 cancelled 的来源路径**
- **位置：** 2.2 状态流转图
- **问题描述：** 状态流转图显示 cancelled 状态存在，但"哪些状态可以流转到 cancelled"仅在表格 2.3 中体现，图中未标注
- **建议：** 在流转图中补充 cancelled 的入边（running/testing 等 → cancelled）

## 三、PRD-产物交付.md 评审

### 🔴 严重问题

**I-10: M-5"三端一致"定义不明确**
- **位置：** Must 验收标准 M-5
- **问题描述：** 验收标准要求"回退后代码/PRD/预览三端一致"，但 PRD 正文中未明确定义"三端"具体指哪三端（可能是：GitHub 代码、PRD 内容、预览链接指向的部署版本）
- **影响：** 测试断言无法精确编写
- **建议：** 在验收标准中明确定义"三端"的具体检查项（如：GitHub commit hash、PRD version_id、preview_url 指向的 deployment_id）

### 🟡 中等问题

**I-11: E-8 Render 不支持中止的处理方式需进一步澄清**
- **位置：** 4.5 回退异常场景处理 E-8
- **问题描述：** E-8 说明 Render 不支持中止 API，"用户可选择强制中止（标记状态为 `aborted`，不等候）"，但未定义 `aborted` 状态后续如何流转到回退完成
- **影响：** 异常场景测试用例无法完整设计
- **建议：** 补充 `aborted` 状态的定义及其在状态机中的位置

**I-12: S-1 版本对比测试标准需量化**
- **位置：** Should 验收标准 S-1
- **问题描述：** "两个版本间的文本差异（插入/删除）100% 展示在 Diff 视图中"要求 100% 不遗漏，但未定义"遗漏"的判定标准（如：连续多行变更的行号偏差算不算遗漏？）
- **影响：** 测试用例的 pass/fail 判定存在歧义
- **建议：** 补充 Diff 测试的 expected output 示例或定义 diff-match-patch 库的输出格式作为判定基准

**I-13: M-6 失败原因枚举未在 PRD 正文中完整列出**
- **位置：** Must 验收标准 M-6
- **问题描述：** M-6 要求"失败原因枚举明确（网络错误/权限不足/构建失败/超时/未知）"，但此枚举列表未在正文功能设计中明确定义，仅在5.4 加载与错误状态中部分提及
- **影响：** 无法确认枚举是否完整，自动化测试用例可能遗漏边界枚举值
- **建议：** 在 M3 部署控制台或 M6 验收标准附近补充完整的失败原因枚举定义

### 🟢 轻微问题

**I-14: S-4 部署历史记录"刷新页面不丢失"的存储方式需明确**
- **位置：** Should 验收标准 S-4
- **问题描述：** "刷新页面不丢失（localStorage）"隐含了实现细节，但未说明 localStorage 容量限制、超限后的降级策略
- **建议：** 补充超限处理方案（如 LRU 淘汰、只保留最近 10 条）

**I-15: 预览展示方式与全局文档不一致**
- **位置：** M1.1 PRD 展示区 vs 5.3 交互细节
- **问题描述：** M1.1 定义 iframe 内嵌展示（默认），5.3 定义用户提供切换开关改为新标签页，但 PRD-首页.md 中 M3.1 通知渠道提到"点击通知，直接跳转到产物交付页面"，未说明此时默认用哪种展示方式
- **建议：** 统一"通知跳转后"的默认展示方式
