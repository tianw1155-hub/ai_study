### 2.3 仍遗漏的 P0 问题

#### 🔴 P0-STILL-1：所有 LLM 均不可用时无降级策略

**严重程度：** P0

**问题描述：**
第 3.3 节限流策略只处理「单次 429 重试失败」，没有处理「DeepSeek **和** Gemini **都**不可用/限流/配额耗尽」的场景。

当所有 LLM 均不可用时：
- 当前行为：需求永久卡在「LLM 生成中」状态
- 3 次重试后标记失败，但用户无法得知是限流还是模型彻底不可用
- 用户自配 Key 的降级路径没有定义（平台内置 Key 失败 → 是否可切用户 Key？）

**影响：**
- 平台级 LLM 服务异常时，所有用户需求同时卡死
- 用户自配 Key 无法在平台 Key 失败时自动启用

**修复建议：**
在第 3.3 节补充降级链路：

| 层级 | 策略 |
|------|------|
| 单模型 429 | 指数退避 1s→2s→4s，重试 3 次 |
| 单模型彻底失败（403/401/500） | 立即切换同层级备用模型（DeepSeek↔Gemini） |
| 所有平台内置模型不可用 | 提示用户绑定自配 Key，或排队等待 |
| 用户自配 Key 限流 | 由用户自行处理，平台不介入 |

---

#### 🔴 P0-STILL-2：完成标准只有"预览 URL"没有"功能正确性"（P0-NEW-3 仍未解决）

**严重程度：** P0（质量维度）

**问题描述：**
第 2.1 节「可发布」标准只要求「平均交付时间 < 2h/条」，第 5.2 节只验证「预览 URL 返回 200」。

**但：URL 返回 200 ≠ 功能正确。**

常见失败场景：
- 前端页面加载但关键交互按钮 JS 报错
- 后端 API 健康检查通过但核心业务逻辑返回错误数据
- AI 生成的代码符合语法/测试/安全门禁，但与 PRD 描述的需求不符

**影响：**
- 用户看到预览 URL，以为「功能已就绪」，点开发现不对
- 「能预览」≠「功能正确可发布」，两者不等价

**修复建议：**

方案 A（轻量）：在第 5.2 节增加功能验证门槛
> | Vercel 前端 | 预览 URL 返回 200 **且无 JS Error（通过 console.error 检测）** |
> | 后端 | 健康检查通过 **且关键 API 返回预期数据（LLM 自我审查输出）** |

方案 B（完整）：在第 2.1 节「可发布」标准增加功能验证维度
> 10 条需求验证，平均交付时间 < 2h/条，**且功能正确率 ≥ 80%（HiAI review 或用户验证）**

## 五、最终结论

**评审状态：有条件通过（需修复 2 个遗留 P0）**

v0.3 成功修复了上一轮的两个 P0 问题：
- CI 门禁现在包含「所有测试必须 PASS」
- LLM 限流现在有明确的指数退避重试策略

但两个上轮已提出的 P0 问题在本轮仍未解决：
1. **多 LLM 全链路不可用时无降级策略** — 影响平台级容错能力
2. **完成标准只验证 URL 可达，不验证功能正确** — 影响用户实际体验

这两个问题不影响进入详细设计阶段，但建议在最终版 PRD（v1.0）前明确。
