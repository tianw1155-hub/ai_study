## 一、P0 修复验证

### P0-3：重试间隔无量化验收标准

**问题描述（v0.3）：** 重试间隔明确定义为指数退避（10s → 30s → 90s），但 Should 验收标准中无对应测试指标，M-6 仅验证 retryCount +1，无法发现时间间隔不符合预期的问题。

**v0.4 修复情况：**

✅ **已修复** — 任务看板验收标准新增 S-5：

| ID | 标准 | 指标 | 测量方式 |
|----|------|------|----------|
| S-5 | 重试间隔符合指数退避 | 重试触发时间符合指数退避（允许 ±10% 误差） | 记录每次重试的 actual_delay，验证 10s ≤ actual_delay[1] ≤ 11s，27s ≤ actual_delay[2] ≤ 33s，81s ≤ actual_delay[3] ≤ 99s |

**验证结果：** S-5 量化指标完整，测量方式明确。三个退避台阶（10s / 30s / 90s）及容差范围（±10%）均有定义，测试可执行。

**测试建议：** 设计自动化时间戳打点测试，覆盖 fail → retry → fail → retry 的完整链条，断言 actual_delay 在各台阶容差范围内。

### P0-4："三端一致"定义模糊

**问题描述（v0.3）：** 产物交付 M-5 要求"回退后代码/PRD/预览三端一致"，但未明确定义三端具体指哪三端，测试断言无法精确编写。

**v0.4 修复情况：**

✅ **已修复** — 产物交付 M-5 验收标准已明确定义：

> M-5 | 回退功能正常 | 回退后**代码端/PRD 端/预览端三端一致**，更新时间 < 3min | 三端状态逐一验证：<br>- **代码端：** GitHub revert commit SHA 与目标版本 commit SHA 一致<br>- **PRD 端：** 数据库 `current_version_id` 指向目标版本<br>- **预览端：** preview_url 指向的 deployment_id 对应目标版本 commit<br>三个值两两比对一致，才算"三端一致"

**验证结果：** 三端定义清晰，每个端都有明确的检查方式（SHA 比对、version_id 指向、deployment_id 对应）。验收判定逻辑明确（三个值两两一致才通过）。

**测试建议：** 设计自动化比对测试，回退后分别查询 GitHub API、数据库 version_id、Vercel/Render API 的 deployment_id，三值两两比对，断言完全一致。

## 三、回归问题（v0.4 新引入）

### 🆕 P1：首页 M2.3 断线重连描述与其他文档不一致

**位置：** PRD-首页.md → M2.3 WebSocket 实时推送

**问题描述：** 首页 M2.3 连接建立流程中写道：

> 4. 断线重连：指数退避（**1s → 2s → 4s → 8s**），最多重试 5 次

但任务看板 M2.5 重试机制中明确写道：

> 重试间隔：**指数退避：10s → 30s → 90s**

且 S-5 定义的测量标准也基于 10s/27s/81s 的退避序列。两处描述的退避时间**数量级相差巨大**（秒级 vs 十秒级），且首页描述的 1s→2s→4s→8s 实际上是**线性翻倍**，并非指数退避。

**影响：**
1. 测试工程师无法判断实际退避策略是哪个（连接层重连 vs 任务层重试）
2. 首页描述本身存在术语错误（线性翻倍 ≠ 指数退避）
3. 可能导致测试用例编写错误（覆盖错误的退避间隔）

**建议修复：**
- 若 1s→2s→4s→8s 是**连接层**断线重连（WebSocket 网络抖动），需明确标注"此为连接层断线重连，与任务重试机制无关"
- 若首页 M2.3 意在描述任务重试间隔，则应改为"10s → 30s → 90s"，与任务看板保持一致
- 术语修正：若坚持 1s→2s→4s→8s，应标注为"指数加倍（exponential doubling）"，而非"指数退避（exponential backoff）"
