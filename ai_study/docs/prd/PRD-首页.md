## 1. 模块职责

首页是 DevPilot 的核心入口，承担以下职责：

| 职责 | 描述 |
|------|------|
| 需求提交 | 用户通过自然语言或文档上传方式提交开发需求 |
| 实时进度可视化 | 展示 AI Agent 团队处理需求的动态过程，增强用户信任 |
| 结果通知 | 及时触达用户，告知任务状态变更和交付完成 |
| 入口引导 | 导航用户快速访问任务看板和产物交付页面 |

**设计原则**：降低用户门槛，零学习成本提交需求；透明化 AI 工作过程，消除"黑箱"焦虑。

## 3. 功能详细设计

### M1：需求提交入口（自然语言 / 文档上传）

#### M1.1 自然语言输入框

| 属性 | 规格 |
|------|------|
| 位置 | 首页核心区域，视觉焦点 |
| 交互方式 | 支持多行文本输入，按 Enter 换行，Ctrl+Enter / Cmd+Enter 提交 |
| 占位符文案 | "用自然语言描述你的需求，例如：做一个用户登录页面..." |
| 字符限制 | 最少 10 字符，最多 2000 字符 |
| 实时字数统计 | 右下角显示当前/最大字符数 |
| 敏感词检测 | 提交时触发，Go API 内置敏感词库，返回提示"需求包含敏感词，请修改后重试" |

#### M1.2 文档上传区域

| 属性 | 规格 |
|------|------|
| 支持格式 | `.md` / `.docx` / `.pdf` / `.txt` |
| 单文件大小限制 | 最大 10MB |
| 文件数量限制 | 单次最多 5 个文件 |
| 拖拽上传 | 拖拽文件到上传区域，支持高亮反馈 |
| 上传进度 | 显示百分比进度条 |
| 解析结果预览 | 上传完成后，显示提取的关键信息摘要（用户主动关闭或点击区域外收起） |

**文档解析职责划分：**

| 文件类型 | 解析方式 | 解析服务 |
|----------|----------|----------|
| `.md` / `.txt` | 前端直接读取文本内容（FileReader API） | 前端 |
| `.docx` | 上传至后端 Go API，由 Go 库（如 `baliance/gooxml`）解析 XML 结构 | **Go API** |
| `.pdf` | 上传至后端 Go API，由 Go PDF 库（如 `signintech/gopdf` 或调用 Python Agent）提取文本 | **Go API** |

> **说明：** `.docx` 和 `.pdf` 解析统一由后端 Go API 负责，不在前端解析（避免引入重型前端库）。Go API 将解析结果以 JSON 结构返回给前端，前端负责渲染预览。

**解析流程：**
```
用户上传 .docx/.pdf
       ↓
  前端 → POST /api/documents/parse（FormData 上传）
       ↓
  Go API → 解析文档 → 提取文本块/结构化信息
       ↓
  返回 { "summary": "...", "blocks": [...], "raw_text": "..." }
       ↓
  前端渲染预览区
```

### M2：实时动态流（Agent 工具调用记录）

#### M2.1 动态流容器

| 属性 | 规格 |
|------|------|
| 位置 | 首页中部，需求提交入口下方 |
| 最大高度 | Desktop 400px（移动端 200px），超出后垂直滚动 |
| 新消息动画 | 从底部滑入，fade-in，持续 300ms |
| 时间戳 | 每条动态右上角显示相对时间（如"刚刚"、"2 分钟前"） |

#### M2.2 动态消息结构（统一 Schema）

所有 WebSocket 消息统一使用以下 schema，`event` 和 `taskId` 为必填字段，`error` 字段在 `task:failed` 事件中必填：

```json
{
  "event": "task:started | task:progress | task:completed | task:failed | task:heartbeat",
  "taskId": "string（任务唯一标识）",
  "id": "msg_001（消息唯一标识）",
  "timestamp": "2026-03-28T19:30:00Z（ISO 8601 格式）",
  "agent": "Planner | Coder | Tester | Deployer",
  "action": "任务分解 | 代码生成 | 单元测试 | 部署（task:progress 时填写）",
  "detail": "详细描述（可选）",
  "status": "running | success | error（task:progress 时使用）",
  "error": "string（task:failed 时必填，错误描述）"
}
```

> **字段说明：**
> - `event`：事件类型，用于前端路由和渲染逻辑
> - `taskId`：关联的任务 ID，用于状态同步
> - `error`：仅 `task:failed` 事件需要，其他事件可不填或填 `null`
```

#### M2.3 WebSocket 实时推送

**连接方式：**
```
WebSocket URL: wss://api.devpilot.com/ws
认证方式：WebSocket 子协议握手认证（Sec-WebSocket-Protocol 头携带 JWT）
```

**连接建立流程：**
1. 前端登录获取 JWT
2. 建立 WebSocket 连接时，通过 `Sec-WebSocket-Protocol` 头携带 JWT（如 `wss://api.devpilot.com/ws`，协议头 `Sec-WebSocket-Protocol: Bearer <jwt_token>`）
3. 服务器在 WebSocket 握手阶段解析 JWT，认证通过后切换到 `devpilot` 协议开始推送消息
4. 断线重连：指数加倍（exponential doubling，1s → 2s → 4s → 8s → 16s → 32s），最多重试 6 次；**此为 WebSocket 连接层断线重连策略，与任务级重试机制（M2.5 节）相互独立**
5. 心跳：客户端每 30s 发送 Ping，服务端响应 Pong

> **安全说明：** JWT 不再放入 URL query params，避免写入 server access logs、浏览器历史记录和 CDN/代理日志。改用 `Sec-WebSocket-Protocol` 头（HTTP 层面加密，WebSocket 握手后不再暴露）。

**WebSocket 消息事件类型（完整枚举）：**

| 事件类型 | 触发时机 | 消息示例 |
|----------|----------|----------|
| `task:started` | 任务开始执行 | `{"event": "task:started", "taskId": "123", "agent": "Planner", "timestamp": "..."}` |
| `task:progress` | Agent 执行中，工具调用 | `{"event": "task:progress", "taskId": "123", "agent": "Coder", "action": "代码生成", "detail": "...", "status": "running", "timestamp": "..."}` |
| `task:state_changed` | 任务状态流转（Kanban 看板核心事件） | `{"event": "task:state_changed", "taskId": "123", "from_state": "running", "to_state": "testing", "timestamp": "..."}` |
| `task:completed` | 单个子任务完成 | `{"event": "task:completed", "taskId": "123", "agent": "Coder", "status": "success", "timestamp": "..."}` |
| `task:failed` | 单个子任务失败 | `{"event": "task:failed", "taskId": "123", "agent": "Tester", "error": "编译错误", "timestamp": "..."}` |
| `task:heartbeat` | 长时间任务心跳保活 | `{"event": "task:heartbeat", "taskId": "123", "timestamp": "..."}` |

> **说明：** Go API 轮询 Temporal（间隔 3s），在 WebSocket 推送前将 Temporal 的原始事件转换为上述格式，前端只依赖此 JSON 结构。

## 4. 验收标准（MoSCoW + 定量）

### Must（必须有）

| ID | 标准 | 指标 | 测量方式 |
|----|------|------|----------|
| M-1 | 自然语言输入框可正常提交需求 | 提交按钮点击后，2 秒内返回任务 ID | 自动化压测 |
| M-2 | 文档上传支持 md/docx/pdf/txt | 上传 10MB 文件成功，解析时间 < 5 秒 | 按文件类型分别计时 |
| M-3 | 实时动态流正确展示 Agent 工作记录 | **端到端 P95 延迟 < 1s**（见下方定义） | WebSocket 消息时间戳打点 |
| M-4 | 站内通知正确送达 | 任务完成后 3 秒内站内信触达 | WebSocket 消息计时 |
| M-5 | 移动端页面可正常访问 | Chrome/Safari 移动版无布局错乱（元素不重叠，核心按钮可见，无横向滚动） | 真机测试（375px/414px） |

**M-3 端到端延迟定义（重新定义）：**

| 分层 | 指标 | 说明 |
|------|------|------|
| **Temporal → Go API（轮询间隔）** | 3s（固定） | Go API 轮询 Temporal 的间隔（无法优化，是端到端延迟的主要来源） |
| P95 **Go API → 前端网络延迟** | < 1s | WebSocket 消息从 Go API 发出到前端接收的时间（P95） |
| **前端渲染延迟** | < 200ms | 前端接收到消息到渲染完成的延迟 |
| **总端到端延迟（P95）** | < 4.2s | Temporal 状态变化 → 用户看到动态更新的全链路 P95 |

> **测量方式：** Go API 在 WebSocket 消息中携带 `server_timestamp`（即从 Temporal 轮询到的时刻），前端计算 `now - server_timestamp`，统计 P95 值。注意：`server_timestamp` 是轮询到消息的时间，而非 Temporal 状态实际变化的时间（最多存在 3s 的轮询窗口误差）。

## 6. API 接口签名（开发前需对齐）

|------|------|------|
| `/api/requirements/submit` | POST | 提交需求，返回 taskId |
| `/api/documents/parse` | POST | 文档解析（multipart/form-data） |
| `/api/sensitive/check` | POST | 敏感词检测 |
| `/api/stats` | GET | 数据统计（首页底部数字） |
| `/api/templates` | GET | 项目模板列表 |
| `wss://api.devpilot.com/ws` | WebSocket | 实时动态流推送（认证通过 `Sec-WebSocket-Protocol` 头） |
