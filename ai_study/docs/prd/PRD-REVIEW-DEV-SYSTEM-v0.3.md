## 二、上一轮 4 个 P0 复验

### ✅ P0-1：SSE 推送链路 → Go API 轮询 Temporal（每3秒）

**复验结论：可行，可接受**

- Go API 作为 Temporal 的轮询代理，前端通过 WebSocket 接收状态，纯单向数据流，架构简洁
- 每 3 秒轮询在 MVP 单用户场景下延迟可接受
- 相比 SSE 推送链路过 Python Agent → Go → 前端，减少了 Python Agent 的外部依赖

**仍需注意的性能问题（非 P0）：**
- 多用户并发时，Go 实例对每个进行中任务每 3 秒查询一次 Temporal，M 个并发任务 × N 个 Go 实例会产生 M×N 次冗余轮询
- 建议：Go 实例层做任务去重（同任务只在一个 Go 实例内轮询一次），或在 Temporal Activity 完成后主动触发一次推送
- 此问题可在 v0.4 中标注为 Should-fix，不阻塞 MVP 开发

### ✅ P0-2：Master Key 存储 → 改存环境变量

**复验结论：可接受，但需补充约束**

- Master Key 不进 DB，纯在进程启动时从环境变量注入，符合最小暴露原则
- Docker Compose 场景下可通过 `docker secret` 或 `.env` 文件管理

**需要补充的约束（建议写入 PRD）：**
1. Master Key 长度不得低于 32 字符（加密算法要求）
2. Master Key 丢失 = 用户 Key 无法解密，必须重新让用户提供 Key
3. 不得将 Master Key 写死在代码或 Dockerfile 中

### ✅ P0-3：CI 缺测试 PASS → 加"所有测试必须 PASS"

**复验结论：通过**

- CI 门禁表已明确："所有测试必须 PASS"，配合覆盖率 ≥ 60% 要求，约束到位

### ✅ P0-4：LLM 限流无策略 → 新增 429 指数退避重试 3 次

**复验结论：通过**

- 1s → 2s → 4s 指数退避策略合理，覆盖常见限流场景
- 3 次仍失败后标记任务失败 + 通知用户 + 记录错误日志，错误处理闭环完整

## 三、评审重点分析与回答

### 重点1：轮询方案是否可行？有没有更优解？

**结论：可行，当前 MVP 可接受。**

| 方案 | 优点 | 缺点 | 适用阶段 |
|------|------|------|----------|
| **当前：Go 轮询 Temporal（3秒）** | 架构简单，Python Agent 无外部依赖，Temporal 为单一真实源 | 有 0-3 秒延迟，资源浪费（无任务变化时仍轮询） | **MVP** |
| Webhook（Temporal → Go） | 实时，无轮询浪费 | 需要额外部署队列（Redis/RabbitMQ），复杂度高 | 后续迭代 |
| Temporal SDK 内置 Observable 接口（如 Go SDK 的 WorkflowHandle 事件流） | 原生支持，实时性好 | 需要 Python Agent 侧引入 Go SDK 绑定，增加耦合 | v0.5+ |

**建议：** MVP 保持当前轮询方案，但 PRD 中增加一条 "v0.5 优化目标：改为 Webhook 推送，消除轮询开销"。

---

### 重点2：Master Key 放环境变量是否够用？

**结论：MVP 够用，生产环境需加强。**

| 维度 | 评估 |
|------|------|
| **安全（MVP）** | ✅ Master Key 不落盘，不在代码中，仅进程可见 |
| **可用性** | ⚠️ 容器重启需重新注入 Key，需在部署文档中明确说明 |
| **密钥轮换** | ❌ 无自动轮换机制，Key 变更需要所有用户重新提供 Key |
| **灾备** | ❌ Master Key 丢失无法恢复，用户 Key 全部报废 |
| **多实例** | ⚠️ 每个 Go 实例需持有相同 Key，增加了密钥分发复杂度 |

**建议：** PRD 中明确 Master Key 的约束（长度、丢失处理流程），并在 v0.5 路线图中标注接入 Vault 或云 KMS（如 AWS Secrets Manager）的计划。

## 四、新发现的 P0 问题

### 🔴 P0-A（关键）：文档表述矛盾 —— 用户 Key 到底存不存？

**位置矛盾：**
- Section 3.3（LLM 方案）写道："用户 Key 存储在 PostgreSQL（由 Master Key 加密）"
- Section 7（安全与隔离）写道："API Key：用户自配自付，平台不存储"

这两句话**直接矛盾**，属于同一需求的不同表述但得出了相反的结论。

**问题分析：**
- 如果"平台不存储"，则 Section 3.3 中 PostgreSQL 存储 + Master Key 加密的整个设计无效
- 如果存储，则 Section 7 的"平台不存储"描述错误

**修订建议：**
Section 7 的 "平台不存储" 改为：
> **API Key：用户自配自付，平台以 Master Key 加密后存储于 PostgreSQL，用户可随时删除**

并补充：
- Key 用途：仅用于本次需求处理的 LLM 调用
- Key 删除：用户在设置页面可一键删除，删除后平台无法恢复

---

### 🟡 P0-B：HiL 介入机制 —— 阻塞 vs 非阻塞 未定义

**问题：** PRD 定义了 3 次 HiL（Human-in-the-Loop）介入节点：
- HiL #1：方向确认
- HiL #2：部署预览确认
- HiL #3：生产上线确认

但**未定义**：用户在 HiL 节点不响应时，任务如何处理？

**缺失场景：**
1. 用户在 HiL #1 后长时间不确认（10分钟？1小时？），任务挂起还是超时取消？
2. 超时后的行为是什么？（通知用户？自动继续？标记失败？）
3. HiL 是**阻塞式**（任务暂停等待）还是**非阻塞式**（任务继续，HiL 作为独立分支）？

**修订建议：**
在 Section 2.3（需求复杂度处理）或 Section 4 补充：

> **HiL 超时机制：**
> - HiL 为阻塞式介入，用户超过 **30 分钟** 未响应，任务自动暂停
> - 暂停期间用户可随时继续，系统以暂停时刻状态恢复执行
> - 超过 **24 小时** 未响应，任务标记为"用户已放弃"，通知用户
> - 用户可在任何 HiL 节点选择"强制继续"（跳过人工确认，直接执行）

---

### 🟡 P0-C：Go API 调用 Python Agent 服务的通信机制缺失

**问题：** 架构图中 Go 层调用 Python Agent（PM Agent、Dev Agent、Test Agent），但：
- 调用协议未定义（HTTP？gRPC？）
- 超时时间未定义
- 失败重试策略未定义
- Python Agent 服务发现机制未定义（Agent 数量不固定，如何寻址？）

**影响：** 这是整个系统调用链路的核心部分，如果 Go → Python Agent 通信无重试机制，则 Temporal Activity 失败重试无法有效传递到 Agent 层。

**修订建议：**
在 Section 3 补充：

> **Go ↔ Python Agent 通信协议：**
> - 协议：HTTP REST（简化，MVP 够用）
> - 超时：单次请求 60 秒（与 LLM 超时 5 分钟解耦）
> - 重试：HTTP 500/503 时指数退避 1s→2s→4s，共 3 次
> - 服务发现：Python Agent 注册到 PostgreSQL `agent_services` 表（心跳机制），Go API 从 DB 获取可用 Agent 地址
> - 或（更简单）：Python Agent 与 Go 同一容器内通过 localhost HTTP 通信

---

### 🟡 P0-D：WebSocket 认证与连接管理缺失

**问题：** Section 3.4 描述了前端通过 WebSocket 接收状态更新，但：
- WebSocket 连接如何认证？（前端用什么凭证建立连接？）
- 连接断开后如何重连？
- 多标签页同时打开如何处理？
- 未登录用户能否建立 WebSocket 连接？

**修订建议：**
在 Section 3.4 或 Section 7 补充：

> **WebSocket 认证：**
> - 建立连接时携带 GitHub OAuth JWT Token（与 HTTP API 同一套认证）
> - 连接建立时验证 Token 有效性，无效则拒绝连接
> - 前端维护心跳（每 30 秒），超过 60 秒无心跳则断开连接
> - 断开时前端自动重连，重连后请求增量状态（从上次接收的 `task_version` 之后）
> - 多标签页：同一用户的所有标签页共享同一个 WebSocket 连接（前端控制）
