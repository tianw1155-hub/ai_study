## 综合判定

**结论：⚠️ 需要修改**

PRD-002 MVP 定位清晰，MoSCoW 定级合理，整体技术选型（React/Next.js + Go/Gin + Python/LangGraph + PostgreSQL）与 PRD-001 v1.1（Go + Temporal）存在架构路线分歧，需在开发前对齐。另外，PRD 引用的 3 份规则文档（`rules/RULES_HOMEPAGE.md`、`rules/RULES_TASKBOARD.md`、`rules/RULES_DELIVERY.md`）在代码仓库中**不存在**，导致功能范围的实际验收标准缺失，这是最优先需要解决的阻塞性问题。

## 一、技术可行性评审

### 1.1 技术栈（⚠️ 与 PRD-001 路线分歧，需对齐）

| 层级 | PRD-002 MVP | PRD-001 v1.1（已评审） | 评估 |
|------|-------------|------------------------|------|
| 后端 | Go + Gin | Go + Temporal | 分歧点：PRD-001 选 Temporal 作为任务编排引擎，PRD-002 似乎退回纯 Gin REST |
| Agent 层 | Python + LangGraph | Python + LangGraph（Split Tier） | ✅ 一致 |
| LLM | 用户自选（GPT-4/Claude/国产） | LLM Gateway 独立服务 | 分歧点：PRD-002 让用户自己配 API Key，平台本身如何调用 LLM 未说明 |

**核心问题**：PRD-002 §3 没有定义平台侧 LLM 调用路径。用户配了 API Key 之后，平台 Agent（PM/Dev/Test）是用用户的钱调用户的 Key，还是平台有自己独立的 LLM Gateway？

- 如果是前者：**安全与费用隔离**问题没有设计（恶意用户是否可以用他人 API Key 费用？）
- 如果是后者：与 PRD-001 的 LLM Gateway 设计不一致

**建议**：明确"平台 LLM 调用"与"用户 API Key"的边界，是两个独立通道还是共用通道。

### 1.2 前端 Next.js（✅ 基本可行）

Next.js App Router + SSR，GitHub OAuth + WebSocket 实时动态流，技术可行性无障碍。

注意事项：
- 首页 M2 实时动态流（展示所有 agent 工具调用记录）需要 **WebSocket 或 SSE** 推送方案。Go/Gin 标准库没有 WebSocket 支持，需引入 `github.com/gorilla/websocket` 或使用 `github.com/gofiber/contrib/websocket`（如果用 Fiber）。
- 动态流的数据量问题：多 Agent 并发时所有用户的动态混在一起，带宽和数据量可能较大，需要分页或按用户过滤。

### 1.3 Agent 层 Python + LangGraph（⚠️ 可行但 LangGraph 在 MVP 中价值有限）

PRD-002 定义的是线性流水线（PM → Dev → Test），而 LangGraph 的核心能力是**DAG 状态机 + 条件分支**，MVP 阶段用不到。

这与 PRD-001 第一轮评审（DEV.md）指出的问题一致：
> "LangGraph 的核心价值在于 DAG 状态机 + 条件分支，MVP 根本用不到它。"

**建议**：MVP 阶段可以用 LangGraph 做线性状态机（`StateGraph`），但不要把 LangGraph 作为"复杂编排"的预期技术债引入。如果 PRD-002 的 Agent 层未来不需要 DAG 演进，可以直接用 LangChain Runnable 或手写状态机，降低依赖复杂度。

### 1.4 GitHub OAuth 与仓库操作（✅ 风险可控）

GitHub OAuth 登录 + 个人仓库读写，有成熟 SDK（`github.com/google/go-github/v51`）。

关键风险点：
- **Rate Limit**：未在 PRD 中提及，GitHub API 无认证 60 req/h，认证后 5000 req/h，Agent 密集调用需加缓存和批量请求
- **Fork 拦截**（§5）：PRD 说禁止 Fork ai_study 仓库，但 GitHub API 没有原生的"禁止 Fork"配置，需要在应用层做判断（API 层面可以设置 repo visibility 为 internal/private，或在 OAuth scope 限制）
- **Token 撤销**：用户在开发中途撤销 GitHub OAuth Token，任务状态如何处理？需要明确定义

### 1.5 数据库 PostgreSQL（✅ 技术选型正确）

PostgreSQL 相比 SQLite（PRD-001 v1.0 的问题），并发友好，有连接池，是正确选择。但需要注意：
- PRD 没有给出具体 Schema 设计（表结构、索引），建议在规则文档或技术设计文档中补充
- JSONB 字段适合存储 Agent 上下文和执行日志，但需要有明确的字段规范，否则查询困难

## 二、架构设计评审

### 2.1 模块拆分（✅ 清晰合理）

3 个界面（首页 / 任务看板 / 产物交付）的职责划分清晰，Kanban 5 列状态机（M1 of §4）设计合理：

```
待处理 → 进行中 → 等待中 → 已完成 / 失败
```

不允许跨状态跳转是好的约束，减少了状态异常的可能性。

### 2.2 用户隔离策略（✅ 清晰且合理）

§5 的用户隔离策略（个人仓库 only、不支持组织仓库、禁止 Fork ai_study）在 MVP 阶段是正确的安全边界。API Key 用户自配自担，平台不碰钱。

**但有一个遗漏**：平台侧的共享状态（如首页动态流 M2）如何在"用户只能看到自己任务"和"展示所有 Agent 调用记录"之间平衡？§5 说"用户只能看到自己提交的需求相关的任务"，但 §4 首页 M2 说"展示所有 agent 的工具调用记录"——这两个需求存在潜在的信息泄露风险（用户 A 可能通过动态流推断用户 B 的任务内容）。

### 2.3 通知机制（⚠️ MVP 唯一渠道风险高）

MVP 所有通知都走首页横幅（§6），没有飞书/TG 渠道（Will't 级别，合理），但这意味着：
- 用户必须保持页面打开才能看到通知
- 任务执行时间可能较长（LLM 生成 PRD / 代码），用户关闭页面后无法感知完成
- 没有通知持久化（任务失败但用户不在线，只能下次登录才知道）

**建议**：至少在通知横幅之外，增加一个"最近通知"列表（持久化到数据库），用户下次登录可以看到上次离开后发生了什么。

### 2.4 缺少规则文档（❌ 阻塞性问题）

PRD §8 "详细规则索引"引用了以下文档：
- `rules/RULES_HOMEPAGE.md`
- `rules/RULES_TASKBOARD.md`
- `rules/RULES_DELIVERY.md`

**这些文件在代码仓库中不存在。** 没有规则文档，意味着：
- 首页动态流的数据格式、刷新策略、过滤规则无定义
- Kanban 看板的状态流转细节、触发条件无定义
- 产物交付的版本管理策略、回退操作定义无定义

**这是 P0 级别的阻塞问题**：没有规则文档，开发的验收标准就是空的。

## 三、开发成本评估

### 3.2 主要成本风险

## 四、风险点评审

### 4.1 高风险项

| # | 风险 | 严重程度 | 说明 |
|---|------|----------|------|
| R1 | **引用规则文档不存在** | 🔴 高 | 3 份规则文档缺失，验收标准无定义，开发和 PM 会有大量边界争议 |
| R2 | **LLM 输出质量无兜底** | 🔴 高 | PRD 没有定义：当 AI 生成的 PRD 不完整/代码有 bug/测试失败时，系统如何兜底或承认失败 |
| R3 | **平台 LLM 调用路径未定义** | 🔴 高 | 用户自配 API Key，但平台 Agent 如何使用这些 Key（是用户身份还是平台身份？）没有说明 |
| R4 | **"自然语言需求"边界不清** | 🟡 中 | 没有输入约束（如最大长度、必须包含的字段），可能导致解析失败或生成质量失控 |
| R5 | **任务取消机制缺失** | 🟡 中 | 用户在中途想取消任务，Kanban 没有取消/终止状态，Agent 执行中的 LLM HTTP 请求无法中止 |
| R6 | **首页动态流数据量** | 🟡 中 | 多用户并发时动态流数据量可能很大，需要分页、过滤策略，否则页面性能下降 |
| R7 | **通知持久化缺失** | 🟡 中 | 所有通知只在首页横幅，用户离线后无法看到任务完成状态 |

### 4.2 与 PRD-001 的路线分歧风险

PRD-002 MVP 与 PRD-001 v1.1 存在架构路线不一致：
- PRD-001 v1.1 选择了 Go + **Temporal**（任务编排层），已通过开发评审但有 2 个 P0 待修复
- PRD-002 看起来选择了 Go + **Gin**（纯 REST），没有 Temporal

**如果两个 PRD 最终合并**：两套任务编排方案需要统一，否则平台会有两套并行的任务管理逻辑。

**建议**：明确 PRD-002 与 PRD-001 的关系——是独立的 MVP 子项目，还是同一个平台的不同阶段？如果是同一平台，需要在架构上对齐（选 Temporal 还是纯 REST+Gin）。

## 五、具体修改建议

### P0（开发前必须修改）

1. **补充缺失的规则文档**（阻塞性）：`rules/RULES_HOMEPAGE.md`、`rules/RULES_TASKBOARD.md`、`rules/RULES_DELIVERY.md` 三份文档必须存在才能进入开发。这 3 份文档需要定义：
   - 首页动态流的数据格式（JSON Schema）、推送频率、过滤规则
   - Kanban 状态机的完整状态列表、触发条件、允许的流转路径
   - 产物交付的版本命名规范、代码回退的 Git 操作步骤

2. **明确平台 LLM 调用路径**（P0）：PRD §3 和 §4 没有说明平台 Agent（PM/Dev/Test）如何调用 LLM——用用户的 API Key（安全隔离问题）还是平台独立的 LLM 服务（与 PRD-001 LLM Gateway 的关系）？

3. **对齐与 PRD-001 的架构路线**（P0）：PRD-002 的任务编排层是 Gin REST 还是 Temporal？与 PRD-001 v1.1 需要明确合并策略。

### P1（进入 Sprint 前修改）

4. **定义"自然语言需求"的输入约束**：最大字符数、必须字段（标题/背景/目标）、不支持的类型（纯视觉设计/纯数据要求/多系统集成），避免范围蔓延。

5. **增加通知持久化**：首页横幅 + 数据库通知记录（最近 N 条），用户离线后重新登录可以看到历史通知。

6. **增加失败兜底策略**：LLM 生成内容质量不达标时，系统如何处理（请求澄清/标记失败/人工介入）？明确质量门禁。

7. **Sprint 分解**：参考 PRD-001 的 Sprint 规划，给出 MVP 的时间估算（建议至少 2-3 周，含 Agent 层调优）。

### P2（建议修改，不阻断开发）

8. **Kanban 增加"已取消"状态**：处理用户中途取消任务的场景，状态机需要完整。

9. **动态流按用户隔离**：首页动态流（M2）需要过滤逻辑，确保用户只能看到自己任务的 agent 调用记录。

10. **GitHub Rate Limit 处理**：在 §4 或规则文档中明确 GitHub API 调用策略（缓存、批量请求、降级方案）。

## 六、总结

| 维度 | 评分 | 说明 |
|------|------|------|
| 技术可行性 | 7/10 | 整体可行，Go/Gin + Python/LangGraph + PostgreSQL 组合稳健，但 LLM 调用路径未定义是架构级缺口 |
| 架构设计 | 7/10 | 界面拆分清晰，用户隔离策略合理，但通知机制偏弱（仅首页横幅），动态流有潜在信息泄露风险 |
| 开发成本 | 6/10 | 缺少 Sprint 分解，Agent 层 LLM Prompt 调优成本最难预估，Must 级别约 8-11 人天，含 Agent 约 13-19 人天 |
| 风险可控性 | 5/10 | 缺失规则文档是最大阻塞，LLM 输出质量兜底缺失，与 PRD-001 路线分歧需要对齐 |

**最终结论**：⚠️ **需要修改后再评审**

核心阻塞项是 3 份引用的规则文档不存在（P0）以及平台 LLM 调用路径未定义（P0）。这两个问题不解决，开发团队没有明确的验收标准，会产生大量返工。
