## 一、验收标准可测试性评审

### 1.1 Must 等级（必须实现）

| ID | 标准 | 指标 | 可测试性 | 风险 |
|----|------|------|----------|------|
| M-1 | PRD 正确渲染 | Markdown 渲染正确率 > 99%，加载时间 < 1s | ⚠️ 部分可测 | "正确率 > 99%" 需要明确定义：是指语法节点不丢失，还是最终视觉呈现一致？建议拆分为"渲染正确率（语法树）"和"加载时间"两个独立指标 |
| M-2 | GitHub 链接可访问 | 点击后 2s 内跳转至正确的 GitHub 仓库 | ✅ 可测 | 网络延迟可 mock/stub，或用录制替代真实请求 |
| M-3 | 代码目录树正确 | 准确率 100% | ❌ 定义不清 | "与仓库实际内容一致"——前提是测试环境有"标准答案仓库"；需明确：是读 GitHub API 的结果还是本地 mock？ |
| M-4 | 部署预览可访问 | 预览链接 30s 内可访问 | ⚠️ 依赖外部 | Vercel/Render 部署时间不可控，测试环境应使用 mock server 或 pre-built 镜像，否则 CI 不可行 |
| M-5 | 回退功能正常 | 代码/PRD/预览三端一致，更新时间 < 3min | ⚠️ 可测但复杂 | 三端一致性验证需要三套断言，超出单个用例范围；建议拆分为三个独立用例 |
| M-6 | 部署状态正确反馈 | deploying 状态实时更新，失败原因明确提示 | ⚠️ 部分可测 | "实时"（< 2s？）和"明确提示"需要精确定义；建议规定状态更新间隔上限 |

### 1.2 Should 等级

| ID | 标准 | 指标 | 可测试性 | 风险 |
|----|------|------|------|------|
| S-1 | 版本对比功能正常 | Diff 展示正确，变更内容覆盖率 > 95% | ⚠️ 需明确定义 | "变更内容覆盖率"是语义覆盖率还是行级覆盖率？需要标准 Diff 样例 |
| S-2 | 部署日志实时滚动 | 日志延迟 < 2s 更新 | ✅ 可测 | 端到端延迟可测量 |
| S-3 | 单文件预览加载速度 | < 500ms（文件 < 100KB） | ✅ 可测 | 条件清晰，可直接计时 |
| S-4 | 部署历史记录完整 | 保留最近 10 次，刷新页面不丢失 | ✅ 可测 | 可写自动化用例验证 localStorage/数据库记录 |

### 1.3 关键问题汇总

**问题 1：性能指标缺乏测试前提**  
M-1 的"99% 正确率"和 M-4 的"30s 内可访问"均依赖外部服务，CI 环境中无法稳定复现。建议在验收标准中增加**测试环境等效指标**（如 mock 环境下 < 5s）。

**问题 2：部分指标无法自动化判定**  
M-6 "失败原因明确提示"和 S-1 "变更内容覆盖率 > 95%"含有主观判断，自动化用例难以一致性地 pass/fail。建议：
- "明确提示" → 改为枚举类型（如：网络错误/权限不足/构建失败/超时），自动化比对枚举值
- "覆盖率 > 95%" → 提供标准 Diff 样例的 expected output，自动化比对

## 三、功能边界清晰度评审

### 3.1 边界清晰的区域 ✅

- **M1 PRD 版本管理**：版本数量上限（50 个）、自动归档逻辑、回退不影响代码——描述清晰
- **M3 部署配置表**：前端/后端配置项一一对应，参数无歧义
- **M4.3 回退限制**：commit 范围（≤10）、冷却时间（5min）、权限（创建者/管理员）——数值明确

### 3.2 边界模糊/缺失的区域 ❌

**问题 3：M1.4 版本回退与 M4 回退机制描述重叠**

M1.4 描述"PRD 回退（仅回退 PRD，不影响代码）"，M4 又定义了三种回退类型（含代码回退）。两处对"回退"的定义存在交叉但表述不一致：
- M1.4 说"回退影响 = 指定版本设为当前版本，生成新版本记录"
- M4 说"代码回退 = GitHub 代码回退到指定 commit"

需要明确：PRD 回退后是否也生成新的 commit？如果是，则与代码回退的 Git revert 逻辑混在一起了。

**问题 4：M4.2 回退流程图中缺少异常分支**

流程图只展示了成功路径，以下异常场景**完全缺失**：

| 异常场景 | 影响 |
|----------|------|
| Git revert 失败（如冲突） | 是否回滚整个事务？ |
| 部署触发后，部署服务超时 | PRD 和 Git 状态是否需要回滚？ |
| 用户在回退过程中（确认后、完成前）关掉页面 | 是否有幂等保护？ |
| 冷却期内用户通过 API/curl 直接触发 | 如何防御？ |
| 回退超过 10 个 commit 时 | 是拒绝还是分批处理？ |

**问题 5：预览链接指向切换的边界不清晰**

"点击历史记录快速切换预览链接指向"——这是否意味着可以指向任意历史部署，而不需要触发新的回退？如果是，则与回退机制（M4）产生了功能重叠。

**问题 6：导出交付包的边界未定义**

5.5 节提到"导出交付包"，但正文中 M1-M4 均无此功能描述。是否属于本次范围？W-List 也未标注。

**问题 7：部署失败后代码状态**

部署失败时，GitHub 代码是否已经更新？如果已更新，是否需要额外的代码回退入口？

### 4.2 缺失的测试场景（高优先级）

**场景 1：回退过程中网络中断**
- 步骤：用户触发回退 → 确认 → 网络中断
- 期望：系统检测到超时，回退不完整时自动回滚或进入错误状态
- 当前状态：❌ 未描述

**场景 2：回退部分成功**
- 步骤：PRD 更新成功 → Git revert 成功 → 部署触发失败
- 期望：前两步是否需要回滚？"原子事务"的保证范围需要明确
- 当前状态：❌ 事务边界定义不清

**场景 3：并发回退请求**
- 步骤：用户 A 和用户 B 同时对同一版本触发回退（均在冷却期内？）
- 期望：第二个请求被拒绝，还是排队？
- 当前状态：❌ 未描述

**场景 4：回退后再次回退**
- 步骤：v3 → 回退到 v2 → 再回退到 v1
- 期望：时间线如何记录？是否允许？
- 当前状态：❌ 未描述

**场景 5：部署中的版本被回退**
- 步骤：部署正在进行时，用户触发回退
- 期望：是否等待部署完成？还是中止并回退？
- 当前状态：❌ 未描述

**场景 6：回退到已归档版本**
- 背景：超过 50 个版本后，最早的版本被归档到对象存储
- 场景：用户想回退到某个已被归档的版本
- 期望：从对象存储恢复后执行回退，但归档恢复时间未定义
- 当前状态：❌ 未描述

**场景 7：权限变化后的回退**
- 步骤：用户触发回退 → 回退开始 → 用户被移除管理员权限
- 期望：已开始的回退是否继续？权限检查在哪个节点？
- 当前状态：❌ 未描述

## 五、综合建议

### 5.1 高优先级（阻塞测试执行）

1. **明确"原子事务"边界**：M4.2 的回退流程必须是原子性的，但当前对部分失败场景的处理完全缺失。需要补充：
   - 明确哪些步骤在同一事务内
   - 明确部分失败时的 rollback 策略
   - 补充失败后的错误状态定义

2. **补充 GitHub API mock fixture**：M2/M4 大量依赖 GitHub API，测试无法在无 mock 的情况下执行。需要：
   - 标准仓库 fixture（含文件树、commit 历史）
   - 模拟 API 429/500 等异常响应
   - 模拟部署服务超时/失败

3. **澄清 M1.4 与 M4 的回退关系**：避免测试时发现两处描述不一致。

### 5.2 中优先级（影响测试质量）

4. **将性能指标转为可量化标准**：M-1"99% 正确率"、S-1"> 95% 覆盖率"需拆解为可自动化判定的规则。

5. **补充部署失败的代码状态定义**：部署失败后代码是否回退、保留还是未知状态。

6. **补充归档版本的回退路径**：超过 50 个版本后的归档回退流程。

### 5.3 低优先级（体验/文档优化）

7. **统一版本号命名规则**：US-2 说"v2"，M1 说"v3（当前）"，需确认是内部版本号还是用户可见标签。

8. **补充导出交付包功能描述**：5.5 节入口对应的功能规格正文缺失。

| 维度 | 评分 | 说明 |
