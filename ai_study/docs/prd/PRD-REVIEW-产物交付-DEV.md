## 一、技术可行性评估

### 1.1 GitHub API 集成

| 功能点 | 实现方式 | 难度 | 风险 |
|--------|----------|------|------|
| 仓库信息卡片 | GitHub REST API (`/repos/{owner}/{repo}`) | 低 | 无 |
| 文件目录树 | GitHub Tree API (`/git/trees/{tree_sha}?recursive=1`) | 低 | 无 |
| 单文件预览 | GitHub Contents API (`/repos/{owner}/{repo}/contents/{path}`) | 低 | 大文件（>1MB）需特殊处理 |
| Commit 历史 | GitHub Commits API (`/repos/{owner}/{repo}/commits`) | 低 | 无 |
| 代码回退 (Git revert) | GitHub REST API (`POST /repos/{owner}/{repo}/git/refs`) | 中 | 需处理 ref 管理 |

**结论：** GitHub 侧技术可行性良好，所有功能均有成熟 API 支持。

### 1.2 Vercel 集成

| 功能点 | 实现方式 | 难度 | 风险 |
|--------|----------|------|------|
| 触发预览部署 | Vercel API (`POST /v13/deployments`) | 中 | 需配置 Vercel Token |
| 获取部署状态 | Vercel API (`GET /v13/deployments/{id}`) | 低 | 无 |
| 获取预览 URL | Vercel API 返回 `url` 字段 | 低 | 无 |
| 实时日志 | Vercel 提供部署日志 API | 中 | 日志获取需轮询，建议结合 WebSocket |

**技术说明：** Vercel 与 GitHub 深度集成，Dev Agent 将代码推送到 GitHub PR 后，Vercel 可自动触发构建。平台 Ops Agent 只需调用 Vercel API 获取部署结果和预览 URL。

**结论：** 可行，官方 SDK (`vercel` npm 包) 可用。

### 1.3 Render / Fly.io 集成

| 功能点 | 实现方式 | 难度 | 风险 |
|--------|----------|------|------|
| 触发后端部署 | Render API (`POST /services/{id}/deploys`) | 中 | 需配置 Render API Key |
| 健康检查 | Render Blueprints 或 Webhook | 中 | 后端需提供健康检查端点 |
| 冷启动时间 | Render 免费版约 30-60s | 中 | 免费版有休眠机制 |

**结论：** 可行，Render API 文档完善。但需要注意 Render 免费版的冷启动延迟和休眠限制。

### 1.4 高风险项

| 高风险项 | 描述 | 建议 |
|----------|------|------|
| **原子事务回退** | M4.2 回退流程要求"原子事务"同时更新：PRD 当前版本 + Git revert + 触发重新部署。但三者之间没有真正的原子性保障——如果 Git revert 成功但部署失败，系统会处于不一致状态。 | 设计补偿机制（compensating transaction）：记录回退状态，每步操作结果写入 `rollback_logs` 表，支持断点续做。 |
| **实时日志推送** | PRD 中要求"日志延迟 < 2s 更新"，但未说明推送机制。Vercel/Render 的日志 API 均为拉取模式。 | 使用 Go 端轮询 + WebSocket 推送，或使用 Vercel/Render 的 webhook 回调。建议在 SYSTEM PRD 的 WebSocket 架构中统一设计。 |
| **Diff 对比算法** | 版本对比功能（按文件/段落/关键词）需要前端实现 diff 算法。 | 建议使用成熟库如 `diff` (npm) 或 `jsdiff`，不要自研。 |

## 二、验收标准评估

### 2.1 Must（必须有）

| ID | 标准 | 定量指标 | 可测性 | 评估 |
|----|------|----------|--------|------|
| M-1 | PRD 正确渲染 | Markdown 渲染正确率 > 99%，加载时间 < 1s | **难量化**。"正确率"如何测量？建议改为：所有标准 Markdown 语法（含 GFM）渲染正确，加载时间有 SLO 保证。 | ⚠️ 需细化 |
| M-2 | GitHub 链接可访问 | 点击后 2s 内跳转至正确的 GitHub 仓库 | 可测（手工点击验证） | ✅ 可接受 |
| M-3 | 代码目录树正确 | 文件列表与仓库实际内容一致，准确率 100% | 可测（对比 GitHub 仓库实际文件列表） | ✅ 可接受 |
| M-4 | 部署预览可访问 | 部署完成后，预览链接 30s 内可访问 | 可测（计时器测量） | ⚠️ "30s 内可访问"需要明确定义"可访问"是 HTTP 200 还是首屏加载完成 |
| M-5 | 回退功能正常 | 回退后代码/PRD/预览三端一致，更新时间 < 3min | 可测，但"三端一致"需要三方同时验证 | ✅ 可接受 |
| M-6 | 部署状态正确反馈 | deploying 状态实时更新，失败原因明确提示 | 可测 | ✅ 可接受 |

**M-1 改进建议：**
```
原：Markdown 渲染正确率 > 99%，加载时间 < 1s
改为：标准 GFM Markdown（含表格、代码块、任务列表）渲染正确率 = 100%，P95 加载时间 < 1s
```

### 2.2 Should（应该有）

| ID | 标准 | 定量指标 | 可测性 | 评估 |
|----|------|----------|--------|------|
| S-1 | 版本对比功能正常 | Diff 展示正确，变更内容覆盖率 > 95% | **难以量化**。"覆盖率"没有明确定义。 | ⚠️ 需重新定义 |
| S-2 | 部署日志实时滚动 | 日志延迟 < 2s 更新 | 可测，但需要明确端到端链路。 | ✅ 可接受，需配合 WebSocket 实现 |
| S-3 | 单文件预览加载速度 | < 500ms（文件 < 100KB） | 可测 | ✅ 可接受 |
| S-4 | 部署历史记录完整 | 保留最近 10 次记录，刷新页面不丢失 | 可测 | ✅ 可接受 |

**S-1 改进建议：**
```
原：Diff 展示正确，变更内容覆盖率 > 95%
改为：两个版本之间的文本差异（插入/删除）100% 展示在 Diff 视图中
```

### 2.3 总体评价

**优点：**
- 大部分验收标准有明确的定量指标
- 覆盖了 Must/Should/Could/Won't 分级，结构清晰
- 时间指标（如 30s、3min、500ms）具体可测

**问题：**
- 部分"率"类指标（M-1 正确率、S-1 覆盖率）没有定义测量方法，无法落地
- 建议将"率"改为"完全通过/不通过"类指标

## 四、与 SYSTEM PRD 技术栈一致性

### 4.1 一致性检查

| 技术点 | SYSTEM PRD 规定 | PRD-产物交付 描述 | 是否一致 |
|--------|-----------------|-------------------|----------|
| 前端框架 | React / Next.js | React / Next.js | ✅ 一致 |
| 后端语言 | Go + Gin | 未明确（从 SYSTEM PRD 推断为 Go） | ✅ 一致 |
| 代码存储 | GitHub | GitHub | ✅ 一致 |
| 前端预览部署 | Vercel | Vercel | ✅ 一致 |
| 后端预览部署 | Render / Fly.io | Render / Fly.io | ✅ 一致 |
| WebSocket 推送 | 有（Go API 轮询 Temporal + WebSocket 推送前端） | 未明确（但 S-2 要求日志实时滚动，隐含需要推送机制） | ⚠️ 需补充 |
| 任务编排 | Temporal | 未明确提及 Ops Agent 由谁触发 | ⚠️ 需补充 |

### 4.2 不一致 / 需澄清项

| 项 | 问题 | 建议 |
|----|------|------|
| **M2.1 分支信息** | PRD 中写"默认分支（main）+ 最新 commit hash"，但未说明若用户仓库默认分支不是 main 如何处理。GitHub 新建仓库默认分支可以是用户自定义。 | 补充：从 GitHub API 获取仓库默认分支名，不要硬编码为 "main" |
| **M1.2 对象存储** | PRD 提到"超过 50 个版本时，最早版本自动归档至对象存储"，但 SYSTEM PRD 未提及对象存储选型。 | 建议：使用 GitHub 本身存储（归档到仓库的 `docs/archive/` 目录），避免引入额外存储服务 |
| **M4.2 原子事务** | SYSTEM PRD 中没有数据库事务设计，但 M4.2 回退流程依赖"原子事务"。 | 建议使用补偿事务模式，参考 1.4 节 |
| **M4.3 回退限制** | "同一版本 5 分钟内不可重复回退"——这个限制在代码层面如何实现？ | 建议在 PostgreSQL 中记录 `last_rollback_at` 字段，前端 + 后端双重校验 |
| **C-1 PR 自动创建** | Won't 列表中说"生产环境部署（非预览）"，但未说明预览部署是否自动创建 PR | 明确：代码回退时是否自动创建 PR？建议回退时创建新 commit 而不是 force-push，保护仓库历史 |

### 4.3 缺失的系统集成说明

以下 SYSTEM PRD 中定义的能力，在 PRD-产物交付 中未体现，需要补充：

| 缺失项 | SYSTEM PRD 描述 | 产物交付 PRD 中应体现 |
|--------|-----------------|----------------------|
| **Ops Agent 触发机制** | "Ops Agent 触发 Vercel/Render 预览部署" | 需要说明产物交付页面如何与 Ops Agent 交互（Temporal Task 触发 vs 手动触发） |
| **WebSocket 状态推送** | Go API 轮询 Temporal + WebSocket 推送 | 部署状态实时更新（S-2）应复用 SYSTEM PRD 的 WebSocket 机制 |
| **HiL #2 确认部署预览** | HiL #2 在测试完成后、部署预览前触发 | 产物交付页的"部署预览"与 HiL #2 的关系需要说明（是否需要 HiL #2 确认后才能部署？） |

## 五、总结

### 5.1 评审结论

| 维度 | 评级 | 说明 |
|------|------|------|
| 技术可行性 | 🟡 基本可行，有 3 个高风险项 | GitHub/Vercel/Render API 均可行，但原子回退和实时日志推送需要额外设计 |
| 验收标准 | 🟡 大部分可测，2 处需细化 | M-1/S-1 的"率"类指标需要重新定义或给出测量方法 |
| 页面布局和交互 | 🟢 整体合理，有 2 处需明确 | 布局结构清晰，缺少预览内嵌 vs 新标签页的明确定义 |
| 技术栈一致性 | 🟡 基本一致，有 4 处需澄清 | 分支名硬编码、对象存储、回退事务、WebSocket 复用需要对齐 |

### 5.2 必须修改项（P0）

1. **M4.2 原子回退设计**：替换为补偿事务模式，补充 `rollback_logs` 表设计
2. **M1.2 版本归档存储**：改用 GitHub 存储归档版本，避免引入对象存储
3. **M2.1 分支信息**：从 GitHub API 动态获取默认分支，不要硬编码 "main"
4. **S-2 实时日志推送机制**：明确使用 WebSocket，复用 SYSTEM PRD 的推送架构
5. **M-1/S-1 指标定义**：将"率"类指标改为明确通过/不通过的判定标准

### 5.3 建议修改项（P1）

1. **M4 部署与 HiL #2 关系**：明确是否需要 HiL #2 确认后才能触发部署
2. **预览展示方式**：明确内嵌 iframe 还是新标签页，或提供用户切换选项
3. **C-1 PR 自动创建**：明确回退时是否创建 PR（建议创建 PR 保护仓库历史）
4. **复制按钮反馈**：补充 Toast 提示设计

### 5.4 PRD 自洽性问题

| 问题 | 位置 | 说明 |
|------|------|------|
| 版本归档位置 | M1.2 vs 5.5 | M1.2 说归档到对象存储，5.5 快捷操作"导出交付包"又说下载 zip 包，自相矛盾 |
| 预览链接展示 | US-3 vs 5.4 | US-3 说"新标签页打开预览地址"，5.4 又说"Iframe 显示"，需统一 |
| 回退影响 PRD vs M4 | M1.4 vs M4.1 | M1.4 说"仅回退 PRD 内容，不影响代码"，但 M4.1 的回退类型表格中"代码回退"也会影响代码，存在歧义 |
