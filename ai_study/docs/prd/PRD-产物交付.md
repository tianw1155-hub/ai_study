## 1. 模块职责

产物交付是 DevPilot 全流程的终点模块，承担以下职责：

| 职责 | 描述 |
|------|------|
| PRD 版本管理 | 记录需求文档的版本变更，支持历史追溯和版本对比 |
| 代码交付 | 聚合所有生成的代码，提供 GitHub 仓库访问 |
| 一键部署预览 | 支持前端 Vercel、后端 Render/Fly.io 自动化部署 |
| 回退机制 | 支持部署版本回退，保障线上稳定性 |

**设计原则**：交付物可查、可试、可回；用户无需配置即可获得可运行产物。

|------|------|

**版本归档到 GitHub 的实现：**
```
触发条件：版本数量 > 50
操作：
  1. 将最早的版本内容写入 docs/archive/v{version}_{timestamp}.md
  2. Git add + commit + push
  3. 数据库中标记该版本状态为 "archived"，记录 archive_commit_sha
回退到归档版本时：
  1. 从 GitHub docs/archive/ 目录读取对应文件内容
  2. 恢复为当前版本（新建版本记录，不修改历史）
```

### M4：回退机制

#### 4.1 回退类型

| 类型 | 触发条件 | 影响范围 |
|------|----------|----------|
| PRD 回退 | 用户手动触发 | 仅 PRD 内容回退 |
| 代码回退 | 用户手动触发 | GitHub 代码回退到指定 commit（**创建新 revert commit，不 force-push，保护仓库历史**） |
| 部署回退 | 用户手动触发 | 预览链接指向历史部署版本 |

#### 4.2 补偿事务回退流程

> **说明：** 由于 Git revert、数据库更新、部署触发三者无法保证原子性，采用**补偿事务模式**（compensating transaction）。每步操作结果写入 `rollback_logs` 表，支持断点续做。

**`rollback_logs` 表设计：**

| 字段 | 类型 | 说明 |
|------|------|------|
| id | UUID | 回退记录唯一标识 |
| task_id | VARCHAR | 关联任务 ID |
| target_version | VARCHAR | 回退目标版本 |
| step | INT | 当前步骤编号（1-4） |
| step_name | VARCHAR | 步骤名称 |
| status | ENUM | pending / completed / failed |
| error_message | TEXT | 失败原因（如有） |
| created_at | TIMESTAMP | 步骤开始时间 |
| updated_at | TIMESTAMP | 步骤完成时间 |
| completed_at | TIMESTAMP | 整个回退完成时间 |
| retry_count | INT | 当前重试次数（每次断点续做 +1，上限 3 次） |
| github_revert_sha | VARCHAR | Git revert 产生的 commit SHA |
| deployment_id | VARCHAR | 回退触发的部署 ID（用于中止部署） |
| last_rollback_at | TIMESTAMP | 最近一次回退触发时间（用于冷却期校验） |

**回退步骤（调整后顺序 — 先 Git revert 再更新数据库）：**

| 步骤 | 操作 | rollback_logs 记录 | 补偿操作（失败时） |
|------|------|---------------------|-------------------|
| 1 | **Git revert commit（GitHub API）** | step=1, status=completed, `github_revert_sha` 填充 | 若 Step 2 失败：手动 `git revert` 回滚已创建的 revert commit |
| 2 | 更新 PRD 当前版本（数据库） | step=2, status=completed | 回滚 version_id |
| 3 | 触发重新部署（Vercel/Render API） | step=3, status=completed, `deployment_id` 填充 | 记录失败，保留 step 1-2 状态，提供"继续部署" |
| 4 | 回退完成通知用户 | step=4, status=completed | - |

> **顺序调整原因：** 若先更新数据库（Step1），再 Git revert（Step2），Step2 失败后 Step1 无法回滚（补偿事务失效）。调整为"先 Git revert 再更新数据库"后，Git revert 失败时只需废弃 revert commit，数据库无副作用。

**回退流程图（补偿事务模式）：**

```
用户点击"回退到此版本"
        │
        ▼
   ┌─────────┐
   │ 确认弹窗 │
   └────┬────┘
        │ 用户确认
        ▼
   ┌─────────────────────────────────────────┐
   │  Step 1: Git revert commit（先做）      │
   │  POST /repos/{owner}/{repo}/git/refs   │
   │  写入 rollback_logs: step=1,            │
   │  github_revert_sha 填充                 │
   └────┬────────────────────────────────────┘
        │ 成功
        ▼
   ┌─────────────────────────────────────────┐
   │  Step 2: 更新 PRD 当前版本（数据库）     │
   │  写入 rollback_logs: step=2            │
   └────┬────────────────────────────────────┘
        │ 成功
        ▼
   ┌─────────────────────────────────────────┐
   │  Step 3: 触发重新部署                   │
   │  POST /v13/deployments                 │
   │  写入 rollback_logs: step=3,           │
   │  deployment_id 填充                     │
   └────┬────────────────────────────────────┘
        │ 成功
        ▼
   ┌─────────────────────────────────────────┐
   │  Step 4: 回退完成通知                   │
   │  写入 rollback_logs: step=4, completed  │
   │  last_rollback_at 更新                 │
   └────┬────────────────────────────────────┘
        │
        ▼
   ┌─────────┐
   │ 回退完成 │
   │ 通知用户 │
   └─────────┘
```

**断点续做机制：**
- 用户可在"我的任务"页面查看回退记录状态
- 失败的回退可点击"继续回退"从断点恢复
- 每个步骤最多重试 3 次，超过后标记为失败需人工介入

#### 4.3 回退限制

| 限制项 | 值 |
|--------|-----|
| 单次回退 commit 范围 | 最多回退 10 个 commit（边界值：10 可回退，11 拒绝） |
| 回退冷却时间 | 同一版本 5 分钟内不可重复回退（前端 + 后端双重校验） |
| 回退权限 | 仅任务创建者和管理员可操作 |
| 并发回退 | 同一任务同一时间只允许一个回退操作（数据库行锁） |

**冷却时间实现：**
```sql
-- PostgreSQL: tasks 表新增 last_rollback_at 字段
ALTER TABLE tasks ADD COLUMN last_rollback_at TIMESTAMP;
-- 校验：回退前检查 last_rollback_at + 5min > now()
```

#### 4.4 回退通知

| 通知时机 | 通知内容 |
|----------|----------|
| 回退开始 | "正在回退到 v2 版本，请稍候..." |
| 回退进行中 | "回退步骤 {n}/3 进行中..." |
| 回退完成 | "已成功回退到 v2，预览链接已更新" |
| 回退失败 | "回退失败，原因：[{错误类型}]，请稍后重试或联系管理员" |

#### 4.5 回退异常场景处理（完整 7+2 场景）

| # | 异常场景 | 处理方式 |
|---|----------|----------|
| E-1 | **Git revert 失败（如冲突）** | rollback_logs step=1 标记 failed，发送通知"回退失败：Git 冲突，请手动处理"，停止后续步骤 |
| E-2 | **部署触发后超时（> 3min）** | rollback_logs step=3 标记 failed，前端提示"部署超时"，提供"重新部署"按钮 |
| E-3 | **用户在回退过程中关闭页面** | 回退继续在服务端执行，用户可从"我的任务"查看进度；完成后发送通知 |
| E-4 | **冷却期内用户通过 API 直接触发** | 后端校验 `last_rollback_at`，返回 429 Too Many Requests，前端显示"请等待 5 分钟" |
| E-5 | **回退超过 10 个 commit** | 前端禁止点击"回退"按钮，后端二次校验并返回 400 Bad Request |
| E-6 | **并发回退请求（两人同时点击）** | 数据库行锁，第一个请求获得锁后执行，第二个请求返回 409 Conflict |
| E-7 | **回退到归档版本（> 50 版本）** | 从 GitHub docs/archive/ 目录恢复文件内容，继续回退流程 |
| E-8 | **部署中的版本被回退** | **Vercel**：支持中止 API `DELETE /v13/deployments/{id}`，调用中止后等待状态变为 `cancelled` 再继续回退流程；**Render**：不支持中止 API，提供"部署进行中，请等待完成后回退"提示，用户可选择强制中止（标记状态为 `aborted`，不等候）
| E-9 | **回退过程中用户权限变更（被移除管理员）** | 已开始的回退继续执行（以发起时权限为准）；权限检查在回退开始前执行 |

## 4. 验收标准

### Must（必须有）

| ID | 标准 | 指标 | 判定方式 |
|----|------|------|----------|
| M-1 | PRD 正确渲染 | **标准 GFM Markdown（含表格、代码块、任务列表）渲染正确率 = 100%**，P95 加载时间 < 1s | 自动化渲染测试：GFM 各语法节点逐一验证 |
| M-2 | GitHub 链接可访问 | 点击后 2s 内跳转至正确的 GitHub 仓库 | 手工点击验证 |
| M-3 | 代码目录树正确 | 文件列表与仓库实际内容一致，**100% 匹配** | 对比 GitHub API 返回的 file list 与前端展示 |
| M-4 | 部署预览可访问 | 部署完成后，预览链接 **HTTP 200 在 30s 内返回** | 自动化计时 + HTTP 状态码断言 |
| M-5 | 回退功能正常 | 回退后**代码端/PRD 端/预览端三端一致**，更新时间 < 3min | 三端状态逐一验证：<br>- **代码端：** GitHub revert commit SHA 与目标版本 commit SHA 一致<br>- **PRD 端：** 数据库 `current_version_id` 指向目标版本<br>- **预览端：** preview_url 指向的 deployment_id 对应目标版本 commit<br>三个值两两比对一致，才算"三端一致" |
| M-6 | 部署状态正确反馈 | deploying 状态实时更新，**失败原因枚举明确**（网络错误/权限不足/构建失败/超时/未知） | 枚举值逐一断言 |

### Should（应该有）

| ID | 标准 | 指标 | 判定方式 |
|----|------|------|----------|
| S-1 | 版本对比功能正常 | **两个版本间的文本差异（插入/删除）100% 展示在 Diff 视图中**，不遗漏 | 使用标准 Diff 样例 expected output 比对 |
| S-2 | 部署日志实时滚动 | 日志延迟 < 2s 更新（Go 端轮询间隔 1s + 网络裕量） | 端到端时间戳打点测量 |
| S-3 | 单文件预览加载速度 | < 500ms（文件 < 100KB） | 自动化计时（> 100KB 文件不适用此指标） |
| S-4 | 部署历史记录完整 | 保留最近 10 次记录，刷新页面不丢失（localStorage） | 自动化刷新验证 |

### Could（可以有）

## 6. 与 SYSTEM PRD 的集成说明

| SYSTEM PRD 能力 | 产物交付中如何复用 |
|----------------|------------------|
