# PRD-DevPilot：智能开发平台系统框架 PRD

> 文档状态：草稿 | 版本：v0.4 | 创建日期：2026-03-28

---

## 1. 产品概述

### 1.1 产品名称

**DevPilot**（中文：织翼）

### 1.2 一句话定位

让用户用自然语言提交需求，AI agent 团队自动完成从需求到代码交付的全流程智能开发平台。

### 1.3 核心价值

| 传统模式 | DevPilot |
|----------|----------|
| 需求到上线：3-7 天 | 需求到上线：< 2 小时 |
| 需要开发团队 | 零开发基础也能提交需求 |
| 人工盯进度 | AI 自动完成，无需盯梢 |

### 1.4 目标用户

- **主要用户**：外部决策者（产品经理、创业者、非技术背景创始人）
- **使用门槛**：会写自然语言即可，无需技术背景

## 2. MVP 成功定义

### 2.1 三层验证标准

| 阶段 | 定义 | 验证方式 |
|------|------|----------|
| **能跑** | 1 条需求能完整跑通：提交 → PRD 生成 → 代码产出 → GitHub PR | 内部测试 |
| **好用** | 5 条不同类型需求，自动化率 ≥ 70% | 内部评测 |
| **可发布** | 10 条需求验证，平均交付时间 < 2h/条 | 内部验收 |

### 2.2 量化目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 平均交付时间 | < 2h / 条（简单需求 < 1h） | 计时起点：用户点击「提交」时刻；计时终点：GitHub PR 创建成功 或 预览 URL 生成时刻（以最后完成者为准） |
| 人工介入次数 | ≤ 3 次 / 需求 | HiL #1（方向确认）、HiL #2（部署预览确认）、HiL #3（生产上线确认） |
| 自动化率 | ≥ 70% | 无人介入完成的需求数 / 总需求数 × 100% |

**自动化率计量说明：**

| 介入类型 | 是否计入人工介入 |
|----------|-----------------|
| 提交需求 | ❌ 不算（用户主动行为） |
| HiL #1 确认方向 | ✅ 算 |
| HiL #2 确认部署预览 | ✅ 算 |
| HiL #3 确认生产上线 | ✅ 算 |
| 纯自动流转（PM→Dev→Test→Ops） | ❌ 不算 |

### 2.3 需求复杂度处理

| 场景 | 处理方式 |
|------|----------|
| 简单需求（单表、单 API） | < 1h 完成 |
| 复杂需求（多表、多 API） | 自动拆分为多条子需求，逐步完成 |
| 超时处理 | 单次 LLM 调用超时 5min，自动重试 2 次，仍失败则标记失败并通知用户 |

## 3. 技术架构

### 3.1 整体架构：Split Tier

```
┌─────────────────────────────────────────────────────────────────┐
│                         用户层（Web App）                          │
│              首页 / 任务看板 / 产物交付（3个界面）                  │
└───────────────────────────┬─────────────────────────────────────┘
                            │ HTTPS
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│                      Go + Gin（接入层）                           │
│              HTTP API / WebSocket / 认证 / 中间件                  │
└───────────────────────────┬─────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────────┐
│                  Temporal（任务编排层）                            │
│          任务调度 / 状态持久化 / 失败重试 / DAG 支持                │
└───────┬─────────────────┬─────────────────┬───────────────────────┘
        │                 │                 │
        ▼                 ▼                 ▼
┌──────────────┐  ┌──────────────┐  ┌──────────────┐
│ PM Agent    │  │ Dev Agent    │  │ Test Agent  │
│ (Python)    │  │ (Python)     │  │ (Python)    │
│ PRD 生成    │  │ 代码实现      │  │ 测试生成     │
└──────┬───────┘  └──────┬───────┘  └──────┬───────┘
       │                 │                 │
       └─────────────────┼─────────────────┘
                         ▼
              ┌──────────────────────┐
              │   LLM Gateway       │
              │   用户自配 Key 或    │
              │   平台内置免费模型   │
              │ (DeepSeek/Gemini)  │
              └──────────────────────┘
```

### 3.2 技术选型

| 层级 | 技术 | 选型理由 |
|------|------|----------|
| 前端 | React / Next.js | 成熟生态，SSR 支持 |
| 接入层 | Go + Gin | 高性能 HTTP 路由，团队熟悉 |
| 任务编排 | Temporal | 分布式任务引擎，状态持久化 + 失败重试内置 |
| Agent 层 | Python + LangGraph | LLM 生态丰富，LangGraph 状态机 |
| 数据库 | PostgreSQL | 元数据持久化，支持 JSONB |
| 代码存储 | GitHub | 天然版本管理，代码即交付物 |
| 部署 | Docker Compose | 单机部署，简单可控 |
| 前端预览 | Vercel | GitHub 集成，自动部署预览，**免费** |
| 后端预览 | Render / Fly.io | 容器化部署，提供免费预览额度 |

### 3.3 LLM 方案

**用户自配 Key + 平台内置免费模型：**

| 模型 | 费用 | 适用场景 |
|------|------|----------|
| **平台内置：DeepSeek-V3** | 免费 | 默认选项，通用场景 |
| **平台内置：Gemini-2.0-Flash** | 免费 | 快速响应场景 |
| **用户自配：GPT-4 / Claude** | 用户自付 | 高质量需求（可选） |

**安全设计：**
- 用户 Key 存储在 PostgreSQL（由 Master Key 加密）
- Master Key（加密用户 Key 的主密钥）存储在**环境变量**，不存 DB
- Workflow Activity 执行时：读取 Key → 传给 Agent → 临时变量 → 不落盘
- 每个用户的 Key 仅用于该用户的需求处理

**限流处理策略：**
- LLM API 返回 429（Rate Limit）时，自动执行指数退避重试
- 重试策略：1s → 2s → 4s，共 3 次
- 3 次仍失败：标记任务失败，通知用户，并记录错误日志

### 3.4 实时状态同步

**Go API 轮询 Temporal，前端通过 WebSocket 接收更新：**

```
Python Agent 执行 → Temporal 记录状态
                         ↓
              Go API 轮询（每隔 3 秒）
                         ↓
              WebSocket 推送前端
```

**说明：**
- Go API 每 3 秒轮询一次 Temporal 任务状态
- 状态变化时，通过 WebSocket 推送给前端
- 无需在 Python Agent 侧维护推送链路

**事件类型：**
- `task:started` — 任务开始
- `task:progress` — 任务进度（如：LLM 生成中、代码写入中）
- `task:completed` — 任务完成
- `task:failed` — 任务失败（含错误信息）

**WebSocket 认证与重连：**

| 维度 | 设计 |
|------|------|
| 认证方式 | JWT Token（在 URL 参数中传递） |
| 连接建立 | 前端登录后，通过 GitHub OAuth Token 换取 JWT，建立 WebSocket 连接 |
| 心跳 | 每 30 秒发送 Ping/Pong，保持连接 |
| 断线重连 | 前端自动重连（指数退避：1s → 2s → 4s → 8s，最多 5 次） |
| 重连失败 | 提示用户刷新页面，重新建立连接 |

### 3.5 Go ↔ Python Agent 通信协议

**调用方式：Temporal Activity（gRPC）**

```
Go Temporal Worker → Python Agent 服务（gRPC）
```

| 维度 | 设计 |
|------|------|
| 协议 | gRPC（Protobuf 定义接口） |
| 服务发现 | Python Agent 服务注册到 Consul / 固定地址 |
| 超时 | Activity 执行超时 10min，超时则 Temporal 自动重试 |
| 重试 | Temporal 内置重试（指数退避），最多 3 次 |
| 并发 | 多个 Activity 可并行执行 |

**Activity 接口定义（示例）：**

```protobuf
service AgentService {
  rpc GeneratePRD(PRDRequest) returns (PRDResponse);
  rpc GenerateCode(CodeRequest) returns (CodeResponse);
  rpc GenerateTests(TestRequest) returns (TestResponse);
  rpc DeployPreview(DeployRequest) returns (DeployResponse);
}
```

**Python Agent 服务部署：**
- Docker Compose 中运行多个 Python Agent 容器
- 每个容器处理分配的任务，共享 PostgreSQL 和 Temporal

## 4. 功能范围（MVP）

### 界面一：首页

- **需求提交入口**（自然语言 / 文档上传）
- **实时动态流**（展示所有 agent 的工具调用记录）
- **结果通知**（任务完成 / 失败时弹出提示）

### 界面二：任务看板

- **Kanban 看板**（5 列：待处理 / 进行中 / 等待中 / 已完成 / 失败）
- **状态机流转**（禁止跨状态跳转）
- **任务详情**（执行日志、错误信息、关联产物）

### 界面三：产物交付

- **PRD 版本管理**（所有评审通过版本可查）
- **代码交付**（GitHub 跳转链接 + **一键部署预览**）
- **回退机制**（PRD + 代码同步回退）

### 4.4 Human-in-the-Loop（HiL）行为规范

**HiL 节点定义（MVP 共 3 个）：**

| 节点 | 触发时机 | 用户操作 |
|------|----------|----------|
| HiL #1 | PM Agent 完成 PRD 初稿 | 确认方向，继续/修改/取消 |
| HiL #2 | 测试完成，部署预览前 | 确认部署预览 |
| HiL #3 | 预览验收完成，生产上线前 | 确认生产上线 |

**超时处理策略：**

| 场景 | 处理方式 |
|------|----------|
| 用户在 48h 内响应 | 正常流程，继续或取消 |
| 用户超过 48h 无响应 | 每日发送提醒通知 |
| 用户超过 72h 无响应 | 升级通知（通知平台管理员） |
| 用户选择「取消」 | 任务终止，保留当前产物，记录取消原因 |

**超时是否阻塞：**
- HiL #1（PRD 方向确认）：**阻塞** — 不确认方向，Dev Agent 不开始开发
- HiL #2 / #3（部署/上线确认）：**非阻塞** — 用户超时后可选择继续或取消，平台不强制等待

## 5. 质量门禁

### 5.1 GitHub PR 合入门禁

所有代码合并前，必须通过以下 CI 检查：

| 检查项 | 工具 | 门禁要求 |
|--------|------|----------|
| 语法/lint | ESLint / golangci-lint | 0 错误 |
| 单元测试 | pytest / go test | **所有测试必须 PASS** |
| 单元测试 | pytest / go test | 覆盖率 ≥ 60% |
| 安全扫描 | Bandit / Semgrep | 0 Critical/High 漏洞 |
| 代码格式化 | Prettier / gofmt | 自动修复 |

**执行时机：** GitHub PR 创建 / Push 时自动触发 GitHub Actions

### 5.2 预览部署成功判定

| 平台 | 成功标准 |
|------|----------|
| Vercel（前端） | 预览 URL 生成且返回 200 |
| Render/Fly.io（后端） | 预览 URL 生成且健康检查通过 |
| 失败处理 | 记录错误日志，通知用户，保留代码可下载 |

## 6. 数据设计

### 6.1 数据库（PostgreSQL）

| 表 | 用途 |
|----|------|
| `tenants` | 租户表（预留多租户字段） |
| `users` | 用户表（GitHub OAuth 信息） |
| `requirements` | 需求表（标题、描述、状态、MoSCoW） |
| `tasks` | 任务表（agent 任务状态） |
| `prd_versions` | PRD 版本表 |
| `exec_logs` | 执行日志表 |

**多租户设计：**
- 所有表都有 `tenant_id` 字段（当前固定为 "1"）
- 查询默认带 `tenant_id` 过滤
- 未来扩展多公司时，加上租户隔离逻辑

### 6.2 GitHub 存储

| 内容 | 存储位置 |
|------|----------|
| PRD 正文 | `docs/prd/PRD-{id}.md` |
| 产物代码 | 用户个人仓库 |
| 执行日志 | PostgreSQL（保留 30 天） |

## 7. 安全与隔离

| 维度 | 设计 |
|------|------|
| 登录方式 | GitHub OAuth |
| 用户隔离 | 用户只能看到自己的需求和产物 |
| 仓库权限 | 仅支持个人仓库，不支持组织仓库 |
| Fork 限制 | 禁止 Fork DevPilot 仓库 |
| API Key | 用户 Key 密文存储在 PostgreSQL，Master Key 存环境变量，执行时解密，用完即释放，平台不保留明文 |

## 8. 部署架构

### 8.1 生产环境（Docker Compose 单机）

```
宿主机（1台 4核8G）
├── PostgreSQL（容器）
├── Redis（容器，可选）
├── Go API（容器）
├── Temporal（容器）
└── Python Agent 服务（容器 × N）
```

### 8.2 预览部署

| 类型 | 平台 | 触发方式 |
|------|------|----------|
| **前端预览** | Vercel | GitHub PR 自动触发，提供临时预览域名 |
| **后端预览** | Render / Fly.io | 平台调用 API 触发部署，提供临时预览 URL |

**预览流程：**
```
Dev Agent 生成代码 → Push 到 GitHub PR
    ↓
Ops Agent 触发 Vercel（前端）/ Render（后端）预览部署
    ↓
平台获取预览 URL，展示在产物交付页面
```

**扩展路径：**
- 1 → 多实例：Nginx 负载均衡 + 共享 PostgreSQL
- 当前不需要 Kubernetes

## 10. MoSCoW 定级（MVP 范围）

| 级别 | 功能 |
|------|------|
| **Must** | GitHub OAuth 登录、需求提交（自然语言）、PM Agent 生成 PRD、Dev Agent 生成代码、任务看板（Kanban）、产物交付（PRD + 代码 + **一键部署预览**）、Temporal 任务编排 |
| **Should** | 测试 Agent 生成测试用例、PRD + 代码同步回退、首页实时动态流 |
| **Could** | 飞书/TG 通知（网页展示即可）、多租户 |
| **Won't** | 飞书/TG 通知（网页展示即可）、多租户、Kubernetes |
