## 二、新发现问题

### P1：首页 M2.3 断线重连退避时间与其他文档不一致 🔴

**发现来源：** 测试工程师复审

**问题描述：**
首页 M2.3 写"断线重连：指数退避（1s → 2s → 4s → 8s）"，但任务看板 M2.5 写"重试间隔：10s → 30s → 90s"，两处数量级相差巨大（秒级 vs 十秒级），且首页描述本身存在**术语错误**：1s→2s→4s→8s 是线性加倍（exponential doubling），不是指数退避（exponential backoff）。

**审核意见：**

此 P1 需要判断是**连接层重连**还是**任务层重试**：

- **如果是连接层（WebSocket 底层断线重连）：** 两处不矛盾，各司其职。但需在首页 M2.3 明确注明"此为连接层，与任务重试机制无关"，避免读者混淆。
- **如果是任务层重试描述：** 首页应与任务看板保持一致，改为 10s → 30s → 90s，并修正术语为"指数退避"。

由于 v0.3 修订中已明确"JWT 安全修复"和"任务看板重试间隔"是两套独立机制（前者是 WebSocket 认证，后者是 Temporal 任务重试），**倾向认为两处不矛盾，但首页描述的术语错误需要修正**。

**建议修复（v0.5）：** 首页 M2.3 断线重连说明中增加注释，明确"此为 WebSocket 连接层断线重连策略，与任务级重试机制（M2.5 节）相互独立"，并将"指数退避"改为"指数加倍（exponential doubling）"。
