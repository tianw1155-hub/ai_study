## 二、新发现的问题

### 🔴 P0-1：覆盖率告警阈值与合规门禁分离（高风险）

**严重程度**：P0 - 阻塞性问题

**问题描述**：
Sprint 2 章节中，M5 测试标准定义了两套互斥的覆盖率机制：

1. **M5 测试标准（定量）**：`有效覆盖率 ≥ 70%`
2. **测试报告生成验收标准**：`< 60% 覆盖率告警`

这两者存在逻辑矛盾：
- 告警阈值 60% ≠ 合规阈值 70%
- 当覆盖率为 65% 时：既满足"不触发告警"，又不满足"≥ 70%"——进入灰色地带，无人处理
- **最严重**：当覆盖率为 40% 时，测试仍会通过（只触发告警），但 Critical 安全漏洞却会**阻断合并**

**具体后果**：
```
场景：Test Agent 生成了代码，有效覆盖率 40%，但没有 Critical 安全漏洞
结果：触发告警（60% 以下）→ 团队收到通知 → 但 PR 可以合并
对比：同样 40% 覆盖率 + 1 个 Critical 安全漏洞
结果：PR 被阻断，无法合并
```

**正确做法**（二选一）：
- **方案 A**：统一为 70% 门禁（覆盖率 < 70% → PR 阻断合并）
- **方案 B**：改为分阶段门禁（覆盖率 < 60% → 告警 + 阻断；60%-70% → 仅告警）

**验收标准修复建议**：
```markdown
# 测试报告生成 验收标准修订
旧版：输出覆盖率报告，< 60% 覆盖率告警
新版：覆盖率 < 70% → PR 阻断合并（Critical/High 安全漏洞同理）
```

**状态**：⚠️ 需要修改

### 🟡 P1-1：Sprint 1 集成测试"10 条不同类型需求"定义缺失（中风险）

**严重程度**：P1

**问题描述**：
Sprint 1 验收标准中：
> 10 条不同类型需求，从收到推 PRD，自动化率 ≥ 70%

缺失以下关键定义：
1. **什么叫"不同类型"？** 按意图（new_feature/bug_fix/question）分类？按业务领域分类？还是按代码复杂度？
2. **谁准备这 10 条需求？** 人工准备还是从历史数据抽取？
3. **什么叫"自动化率 ≥ 70%"？** 10 条中 7 条全自动？还是有其他计算方式？

**实际风险**：执行时团队会对此产生分歧，导致验收流于形式。

**建议修复**：
```markdown
# Sprint 1 M1-M3 串联测试 验收标准修订
旧版：10 条不同类型需求，从收到推PRD，自动化率 ≥ 70%
新版：准备 10 条需求样例（new_feature × 6，bug_fix × 2，question × 2），
     自动化率 = 全自动（无人工介入）完成的需求数 / 10 × 100%，
     目标 ≥ 70%（即至少 7 条需求全程自动完成）
```

**状态**：⚠️ 需要修改

### 🟡 P1-2：PRD 验收标准模板中的验收条件仍为非定量（中风险）

**严重程度**：P1

**问题描述**：
PRD 模板（4.2.2 节）中，`## 6. 验收标准` 的示例内容为：
```markdown
## 6. 验收标准
- {可测试的验收条件}
```

这里的"可测试的验收条件"在后续 AI 生成 PRD 时仍然依赖 LLM 自行理解，没有给出定量下限。

**实际风险**：AI 生成的验收条件可能是"用户可以登录系统"（可测试但非定量），而不是"用户使用正确账号密码可在 3 秒内登录成功率 100%"（定量）。

**建议修复**：在 PRD 模板中增加定量验收条件规范：
```markdown
## 6. 验收标准（定量）
格式要求：每条标准需包含「操作 + 预期结果 + 量化指标」
示例：
✅ 正确：用户使用有效账号密码登录，响应时间 < 2s，成功率 = 100%
❌ 错误：用户可以登录系统
```

**状态**：⚠️ 需要修改

### 🟡 P1-3：M5 MVP 范围与 M5 测试标准存在范围不一致（低-中风险）

**严重程度**：P1

**问题描述**：
3.2 节 MVP 范围定义中 M5 Test Agent MVP 范围：
> 基础单元测试生成 + 本地运行

但 6.4 节 Sprint 2 M5 测试标准中已经定义了**完整**的测试标准：
> 有效覆盖率 ≥ 70%、边界用例数、安全扫描（Bandit + Semgrep）、漏洞分级、失效感知

这两者存在矛盾：
- 如果 MVP 只做"单元测试生成 + 本地运行"，则无法覆盖安全扫描（Bandit + Semgrep 需要独立工具链）
- 如果 MVP 就需要满足全套 M5 测试标准，则 3.2 节的 MVP 范围描述不准确

**建议修复**：
3.2 节 M5 MVP 范围修订为：
```markdown
M5: Test Agent | ✅ 单元测试生成 + 本地运行 + 覆盖率门禁（≥70%）|
```
或在 3.2 节增加说明：Sprint 2 才引入安全扫描和漏洞分级。

**状态**：⚠️ 需要修改
