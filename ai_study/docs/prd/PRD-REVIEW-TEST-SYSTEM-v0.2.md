## 一、上一轮 P0 问题修订评审

### P0-1：自动化率无测量定义 ✅ 已解决

**原问题：** 自动化率没有明确的计量方式。

**修订内容（第 2.2 节）：**
> 自动化率 = 无人介入完成的需求数 / 总需求数 × 100%

**评审结论：定义明确，可操作。**

| 介入类型 | 计入人工介入 |
|----------|-------------|
| 提交需求 | ❌ 不算 |
| HiL #1 确认方向 | ✅ 算 |
| HiL #2 确认部署预览 | ✅ 算 |
| HiL #3 确认生产上线 | ✅ 算 |
| 纯自动流转（PM→Dev→Test→Ops） | ❌ 不算 |

**测试工程师评价：**
- 边界清晰，"无人介入"= 0 次 HiL，零歧义
- 表格列举法覆盖了所有常见场景，可直接转化为测试用例的判定条件
- 与「人工介入次数 ≤ 3 次 / 需求」目标兼容，但不矛盾（一个需求可 0 次 HiL，也可以最多 3 次 HiL，但只有 0 次才计入自动化率）

### P0-2：交付时间边界模糊 ✅ 已解决

**原问题：** 交付时间没有明确计时起点和终点。

**修订内容（第 2.2 节）：**
> - 计时起点：用户点击「提交」时刻
> - 计时终点：GitHub PR 创建成功 **或** 预览 URL 生成时刻（以最后完成者为准）

**评审结论：边界清晰，可测量。**

| 边界 | 定义 | 可测试性 |
|------|------|---------|
| 起点 | 用户点击「提交」（前端事件，可埋点） | ✅ 可记录 |
| 终点 | PR 创建 **或** 预览 URL 生成（两者取晚） | ✅ 可记录 |

**补充建议（不影响评审通过）：**
- 建议在数据库 `requirements` 表加 `submitted_at`、`pr_created_at`、`preview_url_at` 三个时间戳字段，方便精确计算
- 「以最后完成者为准」需要系统记录两个时间点后取 max，技术上可实现

---

### P0-3：GitHub PR 质量门禁缺失 ✅ 已解决

**原问题：** 没有 PR 合流前的质量门禁。

**修订内容（第 5.1 节）：**

| 检查项 | 工具 | 门禁要求 |
|--------|------|---------|
| 语法/lint | ESLint / golangci-lint | 0 错误 |
| 单元测试 | pytest / go test | 覆盖率 ≥ 60% |
| 安全扫描 | Bandit / Semgrep | 0 Critical/High 漏洞 |
| 代码格式化 | Prettier / gofmt | 自动修复 |

触发时机：GitHub PR 创建 / Push 时，GitHub Actions 自动执行。

**评审结论：基础门禁已到位。**

## 二、重点问题深入分析

### 2.1 自动化率计量方式合理性评估

**✅ 优点：**
- 定义简洁，易于理解和执行
- 直接与 HiL 次数挂钩，逻辑自洽
- 可追溯：HiL 是用户主动触发的，系统可记录每条需求的 HiL 次数

**⚠️ 潜在风险：**

1. **"完成"的定义依赖结果而非过程**
   - 当前定义只问「有没有 HiL」，不问「HiL 之后是否正确完成」
   - 例：需求自动跑完 HiL#1 → AI 方向错误 → 用户否决重做 → 最终仍有 PR 创建。这个需求算「无人介入完成」吗？
   - 建议：明确定义「完成」的内涵是「PR 创建成功 且预览可用」

2. **没有区分 HiL 类型**
   - HiL#1（方向确认）、HiL#2（预览确认）、HiL#3（上线确认）代价不同
   - 全部 3 次 HiL 的需求 vs. 只做 1 次 HiL 的需求，自动化程度体感差异大
   - 建议考虑分级统计（如：0 HiL / 1 HiL / 2 HiL / 3 HiL 分布）

3. **目标 70% 与「≤ 3 次 HiL」之间的逻辑张力**
   - 如果每条需求最多允许 3 次 HiL，70% 自动化的含义是：30% 的需求至少有 1 次 HiL
   - 这与「AI 生成代码质量还不够高，需要人盯着」的产品现状是匹配的
   - 但 70% 这个数字是怎么得出的？建议补充基准数据或行业对标

**建议测试用例：**
```
Given: 一批 10 条需求
When: 自动化率统计
Then:
  - 0 次 HiL 的需求 × 7 → 自动化率 70%
  - ≥1 次 HiL 的需求 × 3 → 非自动化
  - 验证公式: 7/10 × 100% = 70%
```

### 2.2 CI 门禁设计充分性评估

**✅ 到位之处：**
- lint 0 错误 → 强制代码质量底线
- 覆盖率 ≥ 60% → 防止零测试，量化可测
- 安全扫描 0 Critical/High → 安全红线明确
- 自动格式化 → 减少人工 review 负担

**❌ 缺失的 P0 问题：功能正确性验证**

| 问题 | 说明 |
|------|------|
| **测试必须通过** | 当前只要求「覆盖率 ≥ 60%」，不要求「测试全部 PASS」<br>AI 可能生成通过覆盖率但实际用例全错的测试代码<br>门禁必须包含 `pytest --tb=short` / `go test` 返回码 = 0 |
| **回归测试** | MVP 没有提到现有代码回归测试<br>AI 新增 / 修改代码可能破坏既有功能<br>建议至少跑通现有测试套件 |

**⚠️ 次要缺失：幻觉风险**

AI 生成代码存在「看起来正确但实际不符合需求」的风险，当前门禁组合无法检测：
- 代码功能与 PRD 描述不一致
- API 行为与需求描述不匹配

建议（非 MVP 必需，但应列入后续版本）：
- AI 生成结果自测（LLM 自我审查输出）
- PRD → 代码可追溯性（哪个 PRD 版本生成哪段代码）

**⚠️ 次要缺失：LLM 限流处理**

当 DeepSeek/Gemini 达到 API 限流时：
- 没有降级策略（切换到备用模型？用户自配 Key？）
- 没有限流通知机制

建议：参考「单次 LLM 调用超时 5min，自动重试 2 次」的异常处理模式，补充限流场景处理。

## 三、新发现的 P0 问题

### 🔴 P0-NEW-1：CI 门禁缺少「测试必须通过」要求

**严重程度：** P0

**问题描述：**
第 5.1 节 CI 门禁只要求「覆盖率 ≥ 60%」，但没有要求 `pytest` / `go test` 返回码为 0（全部通过）。

AI 可能生成：
- 通过覆盖率但大部分用例 FAIL 的测试代码
- 空测试函数（满足行覆盖率但不验证任何逻辑）

**影响：**
- 不符合「代码可运行」的基本质量标准
- 预览部署后可能直接报错
- 安全扫描和 lint 过的代码实际功能完全错误

**修复建议：**
在第 5.1 节增加一行：

| 检查项 | 工具 | 门禁要求 |
|--------|------|---------|
| 测试通过 | pytest / go test | 返回码 = 0（全部用例 PASS） |

---

### 🔴 P0-NEW-2：LLM API 限流（Quota）处理策略缺失

**严重程度：** P0

**问题描述：**
第 3.3 节描述了 LLM 选型，但只提到「超时 5min，自动重试 2 次」，没有处理 `429 Too Many Requests` 或配额耗尽的场景。

当 DeepSeek/Gemini API 触发限流时：
- 当前行为未知（重试？降级？挂起？）
- 用户感知不到原因
- 需求可能永久卡住

**影响：**
- 付费 API 配额耗尽后，用户需求无人处理
- 免费 API 限流导致系统不可用
- 没有任何用户通知或降级机制

**修复建议：**
在第 3.3 节或第 2.3 节（超时处理附近）补充：

| 异常类型 | 处理策略 |
|----------|---------|
| LLM 超时（5min） | 自动重试 2 次，仍失败则标记失败并通知用户 |
| LLM 限流（429） | 等待 60s 重试最多 3 次，仍失败则切换到备用模型（DeepSeek↔Gemini）或通知用户 |
| 配额耗尽（403/401） | 通知用户检查 Key 状态，提供用户自配 Key 降级路径 |

---

### 🟡 P0-NEW-3：完成标准只有「预览 URL」没有「功能正确性」

**严重程度：** P0（质量维度）

**问题描述：**
第 2.1 节「可发布」标准只要求「10 条需求验证，平均交付时间 < 2h/条」，没有功能正确性验证。

产品成功不能只靠「有预览 URL」，代码跑通了才算真正的交付成功。

**影响：**
- AI 生成的代码可能 lint/测试都通过，但功能与需求不符
- 「可发布」不等于「功能正确可发布」

**修复建议：**
在第 2.1 节增加功能验证维度：

> 10 条需求验证，功能正确率 ≥ 80%（HiAI review 或用户验证）

或在第 5.2 节「预览部署成功判定」扩展：

| 平台 | 成功标准 |
|------|---------|
| Vercel（前端） | 预览 URL 生成且返回 200 **且页面可交互（无 JS Error）** |
| Render/Fly.io（后端） | 预览 URL 生成且健康检查通过 **且关键 API 返回预期** |

## 五、最终结论

**评审状态：有条件通过（需修复 2 个新增 P0）**

原 v0.2 版本的 3 个 P0 问题已全部解决，CI 门禁基础设计到位，自动化率和交付时间定义清晰。

但 CI 门禁缺少「测试必须通过」这一最基本的功能正确性保障，LLM API 限流处理策略完全缺失，这两项属于新发现的 P0 级别问题，必须在进入详细设计阶段前明确，否则：
- 测试通过但功能错误 → 用户看到的是「能跑但不是我要的」
- API 限流无处理 → 需求卡死，用户不知道原因

建议在 v0.3 中补充上述两个 P0 的解决方案后，进入分模块详细 PRD 阶段。
