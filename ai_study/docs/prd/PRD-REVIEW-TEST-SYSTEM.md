# PRD 评审报告：DevPilot 系统框架

> 评审角色：测试工程师 | 评审日期：2026-03-28 | 评审版本：v0.1

---

## 综合判定

**需要修改**

### 判定理由

PRD 框架层面逻辑通顺，Multi-Agent 分工和技术选型合理，但**关键验收标准和质量门禁缺乏可测试性定义**。多处出现「AI 产出」相关的质量目标（如「PRD 可用」「代码可运行」），却未给出如何量化验证的标准。测试工程师无法据此设计测试用例。

---

## 一、测试覆盖可行性评审

### 1.1 各 Agent 产出物可测试性分析

| Agent | 产出物 | 可测试性 | 问题 |
|-------|--------|----------|------|
| PM Agent | PRD 文档（Markdown） | ⚠️ 部分可测 | PRD 内容正确性、格式规范性可验证；但「需求理解准确度」难以自动化度量 |
| Dev Agent | 代码（GitHub PR） | ✅ 可测 | 代码可静态分析、语法检查、单元测试覆盖、CI 检查 |
| Test Agent | 测试用例代码 | ⚠️ 部分可测 | 测试语法可验证；但测试覆盖充分性难以自动化判断 |
| Ops Agent | 预览部署 URL | ✅ 可测 | URL 可访问性、HTTP 状态码、基础功能冒烟 |

**核心问题：PM Agent 和 Test Agent 的产出物本质上依赖 LLM 生成，其「质量」定义模糊。**

### 1.2 GitHub PR 代码质量验证方案

PRD 中未明确 PR 质量验证流程，建议补充：

- **静态扫描**：ESLint / Pylint / Go vet，LLM 生成代码必须通过
- **测试覆盖**：单元测试覆盖率达到阈值（如 70%）才允许 Merge
- **构建验证**：Preview 环境部署成功才算 PR 完成
- **人工 review**：建议至少 1 次人工 Code Review（对应「人工介入次数 ≤ 3 次」）

**问题：PRD 未定义任何 GitHub PR 合入门禁，仅提到「代码到 GitHub PR」，未定义 PR 质量基线。**

## 二、质量门禁设计评审

### 2.1 MVP 阶段建议门禁

| 门禁 | 当前状态 | 建议 |
|------|----------|------|
| 代码语法检查 | ❌ 未提及 | 必须：LLM 生成代码必须通过语言官方 linter |
| 单元测试 | ⚠️ 提及 Test Agent 但未定义通过标准 | 必须：关键路径单元测试通过率 100% |
| 预览部署成功 | ⚠️ Ops Agent 负责但未定义验收标准 | 必须：预览 URL HTTP 200 + 基础功能可运行 |
| PRD 格式校验 | ⚠️ 未提及 | 建议：PRD Markdown 结构完整性校验 |
| 人工确认节点 | ✅ 已有「≤ 3 次」量化目标 | 需明确：哪 3 个节点需要人工介入 |

### 2.2 自动化率 ≥ 70% 如何测量

**当前问题：自动化率定义不清，无法测量。**

建议明确定义：

```
自动化率 =（完全无需人工介入的需求数） / （总需求数）
```

需补充：
- **人工介入的明确定义**：点击确认按钮 / 填写表单 / 回答澄清问题
- **自动重试不计入人工介入**：LLM 超时重试 2 次属于自动流程
- **测量工具**：在任务看板记录每次人工介入事件，最后统计

## 三、验收标准可测试性评审

### 3.1 「平均交付时间 < 2h」可测试性

| 问题 | 说明 |
|------|------|
| ✅ 可测量 | 时间戳记录：需求提交时间 → PR 创建时间 → 预览部署完成时间 |
| ❌ 未定义计时边界 | 计时从何时开始（用户提交？系统接收？）到何时结束（PR 创建？还是预览部署完成？） |
| ❌ 未定义统计口径 | 「简单需求 < 1h」与「平均 < 2h」是不同统计群体，需分开统计 |
| ❌ 未定义超时失败 | 单次 LLM 调用超时 5min 的需求，时间如何计算？是否计入平均值？ |

**建议**：明确计时边界为「需求提交时刻 → 预览部署 URL 可访问时刻」，并对超时失败任务单独统计。

### 3.2 复杂需求自动拆分可验证性

| 问题 | 说明 |
|------|------|
| ⚠️ 缺少拆分质量标准 | 拆分成多少条子需求算「正确拆分」？按什么维度拆分（功能模块？API？页面？）？ |
| ⚠️ 缺少验证方法 | 自动拆分的子需求，如何验证其完整性和正确性？ |
| ⚠️ 拆分失败策略 | 如果 PM Agent 无法拆分复杂需求，系统如何应对？目前仅说「标记失败并通知用户」，但未定义用户之后的处理流程 |

**建议**：补充「需求可拆分性判定规则」和「人工辅助拆分」流程。

## 四、风险点评审

### 4.1 测试阶段无法控制的质量问题

| 风险 | 级别 | 说明 |
|------|------|------|
| LLM 产出代码的业务正确性 | P0 | 测试阶段无法验证「代码是否实现了需求意图」，只能验证「代码是否按指定方式运行」 |
| PRD 内容与用户原始意图的偏差 | P0 | PM Agent 生成的 PRD 是否真正反映用户意图，无自动化验证手段 |
| Preview 部署平台稳定性 | P1 | Vercel/Render 免费版有调用频率限制，可能导致部署失败，这不是代码质量问题但会影响验收 |
| 多 Agent 协作的端到端一致性 | P1 | PM → Dev → Test → Ops 链路中，前序 Agent 输出格式变化会导致后续 Agent 处理失败 |
| GitHub API 限流 | P1 | 用户操作 GitHub 可能触发限流，导致 PR 创建失败，不属于产品质量问题但影响验收 |
| 数据库 schema 变更 | P2 | Agent 动态生成代码可能修改数据库结构，未提及 schema 迁移机制 |

### 4.2 预览部署验证方案

| 平台 | 验证方法 | 建议 |
|------|----------|------|
| Vercel 前端预览 | 调用 Vercel API 查询 deployment 状态，等待 `READY` 状态，返回预览 URL | 需在 PRD 中明确定义「部署成功」的判定标准（HTTP 200？特定页面元素存在？） |
| Render 后端预览 | 调用 Render API 查询 service 状态，等待 `live` 状态 | 同上 |
| 共同问题 | **预览 URL 有有效期**（Vercel 60 天，Render 免费版可能更短），回退时预览 URL 可能已失效 | 建议在产物交付页面标注预览有效期，并提供手动触发重新部署按钮 |

## 五、具体问题列表

### P0（阻塞发布）

1. **未定义 GitHub PR 质量门禁**
   - LLM 生成的代码没有语法检查和测试覆盖率要求就合入仓库
   - 建议：PR 必须通过 linter + 单元测试 + 至少 1 次人工 review

2. **「自动化率 ≥ 70%」无测量定义**
   - 无法统计就无法验收
   - 建议：明确定义「人工介入」事件及记录方式

3. **「平均交付时间 < 2h」计时边界模糊**
   - 起点终点不清晰，无法统一测量
   - 建议：明确计时从「需求提交时间戳」到「预览 URL 可访问时间戳」

### P1（影响质量）

4. **PM Agent 产出物（PRD）无质量校验机制**
   - PRD 格式完整性、MoSCoW 标注是否规范，均未定义验收标准
   - 建议：定义 PRD 最小结构（背景/功能/非功能/验收标准），并提供格式校验工具

5. **复杂需求自动拆分无质量标准**
   - 如何判断拆分是否正确、完整？人工如何验证？
   - 建议：补充拆分质量自检流程或人工确认节点

6. **Preview 部署「成功」判定标准缺失**
   - URL 返回就代表部署成功？还是需要特定页面内容？
   - 建议：定义最小验证集（如 HTTP 200 + `/api/health` 返回 200）

7. **Ops Agent 预览 URL 有效期未说明**
   - 回退时 URL 失效如何处理？
   - 建议：标注有效期，提供重试部署机制

### P2（建议改进）

8. **人工介入节点「≤ 3 次」未明确具体节点**
   - 建议：在 Kanban 看板的节点处标注（如：需求澄清确认 / PRD 评审确认 / 最终发布确认）

9. **失败任务通知机制过于简单**
   - 失败后「标记失败并通知用户」，用户收到通知后如何操作未定义
   - 建议：定义失败类型（可重试 / 需人工介入 / 需求不合理）和对应处理流程

10. **Test Agent 生成的测试用例质量无法量化**
    - 建议：至少要求生成的测试代码可执行（语法正确 + 可通过 pytest/mocha 等测试框架运行）

## 六、测试建议

### 6.1 MVP 阶段测试策略

| 测试类型 | 覆盖范围 | 自动化要求 |
|----------|----------|------------|
| **单元测试** | 各 Agent 核心逻辑（PM 解析、Dev 代码生成、Test 用例生成） | 必须，自动化 |
| **集成测试** | Agent 间接口（PM→Dev→Test→Ops 链路） | 必须，自动化 |
| **端到端测试** | 完整流程：提交需求 → 获得预览 URL | 必须，自动化 |
| **冒烟测试** | 每次 PR 合入后验证核心流程 | 必须，自动化 |
| **人工测试** | PRD 可读性、代码可维护性、用户体验 | 辅助，非必须自动化 |

### 6.2 关键测试指标

| 指标 | 测量方式 | 目标值 |
|------|----------|--------|
| 端到端成功率 | 成功完成的需求数 / 总需求数 | ≥ 80%（MVP） |
| 自动化率 | 无人工介入的需求数 / 总需求数 | ≥ 70% |
| 平均交付时间 | 所有成功需求的总时长 / 成功需求数 | < 2h |
| Preview 部署成功率 | 部署成功数 / 触发部署数 | ≥ 90% |
| LLM 调用失败率 | LLM 调用异常次数 / 总调用次数 | < 5% |

### 6.3 测试环境建议

- **测试用 GitHub 账号**：隔离的测试组织或个人仓库，避免污染真实代码
- **Mock LLM**：使用记录/回放模式（VCR 或类似工具），避免测试依赖外部 LLM API
- **测试数据**：准备 5-10 条不同复杂度的标准需求（简单/中等/复杂），用于回归验证

---

## 七、总结

DevPilot 系统框架 PRD 整体架构清晰、技术选型合理，但在**质量验收标准**和**可测试性**方面存在明显不足。主要缺失：

1. **测量定义缺失**：自动化率、交付时间等核心 KPI 无明确测量方案
2. **质量门禁不完整**：代码质量、部署成功的验收标准均未定义
3. **失败场景覆盖不足**：复杂需求拆分、预览部署有效期等边界情况未说明

建议在进入各模块详细 PRD（如 PRD-PM-Agent、PRD-Dev-Agent）之前，先补充上述 P0 问题的定义，以确保各模块设计有明确的验收依据。
