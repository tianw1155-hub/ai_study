## 一、上轮遗留问题检查

### 🔴 遗留 #1：多模型降级链路缺失（DeepSeek + Gemini 都挂时无降级）

**状态：未解决 ❌**

**分析：**

Section 3.3「LLM 方案」中，限流处理策略仅描述了同一模型的指数退避重试（429 → 1s → 2s → 4s，3 次），未描述跨模型降级。

当以下情况同时发生时，系统无降级方案：

```
用户未配置自己的 Key（默认情况）
AND DeepSeek-V3 服务不可用
AND Gemini-2.0-Flash 服务不可用
→ 3 次重试全部失败 → 任务标记失败，通知用户
```

**具体问题点：**

1. DeepSeek API 在中国大陆访问受限（需代理），未说明平台是否提供代理或使用官方 API 地址
2. 未定义 Gemini 使用的 API 端点（Google AI Studio / Vertex AI），两者免费额度不同
3. 用户未配置 Key 时，系统完全依赖平台内置模型，但平台模型"全挂"时无兜底
4. 「通知用户」之后，用户如何手动选择模型或重试，无说明

**建议 v0.5 补充：**
- 明确平台内置模型的 API 端点和访问方式（代理/直连）
- 定义降级链路：`DeepSeek-V3 → Gemini-2.0-Flash → (通知用户手动配置 Key)`
- 降级时保留失败上下文，支持用户在配置 Key 后从断点恢复

### 🟡 遗留 #2：预览 URL 200 ≠ 功能正确，无验证

**状态：部分缓解，仍不充分 ⚠️**

**分析：**

Section 5.2「预览部署成功判定」：

> | 平台 | 成功标准 |
> |------|----------|
> | Vercel（前端） | 预览 URL 生成且返回 200 |
> | Render/Fly.io（后端） | 预览 URL 生成且健康检查通过 |

**问题点：**

1. **Vercel 端：** 静态资源 CDN 200 不等于 JS/页面可正常渲染，需至少验证首屏内容或关键 API 调用
2. **健康检查通过 ≠ 功能正确：** Render/Fly.io 的健康检查 endpoint（如 `/health`）仅验证进程存活，不验证业务逻辑
3. **前后端联动验证缺失：** 前端预览依赖后端 API，后端预览依赖数据库，当前标准未覆盖联动场景

**建议 v0.5 补充：**

| 平台 | 建议验证方式 |
|------|-------------|
| Vercel 前端 | 预览 URL 返回 200 **+** 页面 title 非空（fetch 验证） |
| Render/Fly.io 后端 | 健康检查 200 **+** 主动调用 1 个核心 API（如 /api/health 或场景相关接口）验证返回非空 |
| 联动场景 | 前端预览展示"依赖后端 API"时，后端预览必须先完成验证 |

## 二、HiL 超时策略对测试阶段的影响

### 评估：⚠️ 存在测试阶段隐患

Section 4.4 定义了 HiL 超时策略（48h / 72h 升级），但测试阶段需要额外考量。

**问题点：**

#### 1. HiL #1 阻塞导致全链路空转

```
HiL #1（PRD 方向确认）= 阻塞
     ↓
PM Agent 完成 PRD 后，Dev Agent / Test Agent / Ops Agent 全部处于等待状态
     ↓
用户 48h 无响应 → 每日提醒
用户 72h 无响应 → 升级通知平台管理员
```

**测试阶段隐患：** 在内部测试场景中，测试工程师频繁提交需求进行验证，但若每次都需要人工确认 PRD 方向，自动化测试流被频繁中断，无法验证端到端自动化率。

**建议：** 测试阶段提供「HiL 免确认」模式（内部开关），自动跳过 HiL #1/#2/#3，仅在正式用户使用时启用 HiL。

#### 2. 超时机制与 MVP 自动化率目标的冲突

Section 2.2 定义自动化率目标为 ≥ 70%（无人介入完成的需求数 / 总需求数 × 100%），计量方式中 HiL 介入均计入人工。

**冲突：** 若 HiL 超时后系统选择「继续」（Section 4.4 HiL #2/#3 非阻塞），则超时放行业务算作自动化完成，会虚高自动化率，但实际用户体验是"被系统自动推进"而非"真正完成"。

**建议：** 明确超时放行是否计入自动化率，还是单独标记为「超时放行」类别。

## 三、遗漏的 P0 问题

### P0 #1：DeepSeek-V3 / Gemini-2.0-Flash 模型能力边界未定义

**位置：** Section 3.3

**问题：** PRD 未定义各模型在哪些场景下使用，也未说明两者的能力边界差异。

- 复杂需求（多表、多 API 拆分）是否依赖特定模型？
- DeepSeek-V3 与 Gemini-2.0-Flash 的上下文窗口限制？
- 不同模型的工具调用（Function Calling）支持情况？

**影响：** 可能导致部分需求在执行中途因模型能力不足而失败，而非基础设施问题。

**建议：** v0.5 补充模型能力矩阵（上下文窗口、工具调用支持、擅长的任务类型）。

---

### P0 #2：Activity 超时重试与僵尸任务风险

**位置：** Section 3.5，Activity 执行超时 10min，超时 Temporal 自动重试（最多 3 次）

**问题：**

```
Activity 执行 10min 超时
     ↓
Temporal 重试（第 1 次）
     ↓
又超时 10min（第 2 次）
     ↓
再超时 10min（第 3 次）
     ↓
共消耗 30min → 任务最终失败
```

- 30min 的计算资源浪费
- 代码已被部分写入 GitHub PR（若重试发生在代码写入 Activity 中间），产生不完整 PR
- 失败后用户看到的是「任务失败」，而非「部分完成」

**建议：** 对包含写操作的 Activity（如 GenerateCode）拆分得更细，并在 Activity 级别记录幂等性状态，避免重试导致重复写入。

---

### P0 #3：预览部署触发失败后的降级路径

**位置：** Section 5.2，Section 8.2

**问题：** 当前预览部署依赖 Vercel / Render / Fly.io，若：

- Vercel 配额用尽（免费额度达上限）
- Render / Fly.io 部署失败（非 URL 生成失败，而是容器构建失败）
- GitHub PR 因合规问题被 Vercel 拒绝（如仓库名含敏感词）

当前方案只记录错误日志 + 通知用户，但用户收到的通知是「部署失败」，而代码已生成，用户期望的是预览而非自行部署。

**建议：** v0.5 定义降级路径：
1. 主路径：Vercel / Render / Fly.io 预览部署
2. 降级路径：提供代码包下载（ZIP），附带部署说明文档
3. 通知内容需包含失败原因（而非仅「部署失败」）

---

### P0 #4：Temporal 持久化与灾难恢复未定义

**位置：** Section 3.1 / Section 8.1

**问题：** Section 3.1 提到 Temporal 负责「状态持久化」，但未说明：

- Temporal 自身的状态存储（PostgreSQL 还是 Embedded？Section 8.1 未列出）
- Temporal 崩溃后，正在执行的任务如何恢复？
- 宕机重启后，Python Agent 的运行状态如何与 Temporal 同步？

**建议：** v0.5 明确 Temporal 的状态后端（推荐 PostgreSQL + 外挂），并定义灾难恢复流程。

---

### P0 #5：并发需求数量与资源上限未定义

**位置：** 整体架构

**问题：** Section 8.1 定义单机 Docker Compose 部署（4 核 8G），但未定义：

- 同时处理的需求并发上限
- PostgreSQL / Redis 连接池配置
- Python Agent 容器数量 N 的默认值和扩缩容策略
- 超出并发上限后的排队策略（FCFS？优先级？拒绝？）

**测试阶段隐患：** 5 条需求并行测试时可能触发资源瓶颈，导致部分任务超时失败，而非功能问题。

## 四、评审结论

### v0.4 综合评价

| 维度 | 结论 |
|------|------|
| 上轮遗留 #1（多模型降级） | ❌ 未解决 |
| 上轮遗留 #2（预览 URL 验证） | ⚠️ 部分缓解，仍不足 |
| HiL 超时对测试阶段影响 | ⚠️ 存在隐患，需测试阶段开关 |
| 新发现 P0 问题 | 5 个（模型能力边界、Activity 超时僵尸任务、预览部署降级、Temporal 灾备、并发上限） |

### 结论：**需要修改（不阻塞，但建议 v0.5 补齐上述问题）**

### v0.4 可接受的原因

v0.4 相较 v0.3 在以下方面有显著改进：
- HiL 节点定义清晰（3 节点，48h/72h 处理明确）
- Go ↔ Python 调用协议（gRPC）设计合理
- WebSocket 认证与重连机制完整
- PR 合入门禁（CI 强制测试 PASS + 覆盖率 ≥ 60%）

上述 5 个新 P0 和 2 个遗留问题属于**功能边界定义**和**异常路径设计**，不影响 MVP 基本流程跑通，建议在 v0.5 中系统性补齐，不阻塞当前开发启动。
