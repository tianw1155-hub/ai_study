## 一、验收标准可测试性评审

### Must（必须有）- 全部可量化 ✅

| ID | 标准 | 可测试性 | 备注 |
|----|------|----------|------|
| M-1 | 页面加载后 2 秒内完成渲染 | ✅ 可测试 | 自动化 Performance API 可量化 |
| M-2 | 任务状态与列位置一致，准确率 100% | ✅ 可测试 | 数据库状态与 UI 渲染结果逐一比对即可 |
| M-3 | 状态变更延迟 < 500ms | ✅ 可测试 | 前端时间戳 + 后端事件日志可测量 |
| M-4 | 4 个 Tab 全部可切换，内容正确 | ⚠️ 部分模糊 | "内容正确"定义缺失——正确性的判断标准是什么？日志最大条数限制？空状态处理？ |
| M-5 | 取消后任务进入 cancelled 列，更新 < 1s | ✅ 可测试 | 取消操作 + 计时器 + UI 断言 |
| M-6 | failed 任务重试后重新入列 running | ⚠️ 有歧义 | "重新入列"是指回到 Kanban 的 COL_2（处理中）列吗？重试后 `retryCount` 是否 +1？重试次数上限（3次）是否硬性校验？这些细节缺失 |

### Should（应该有）- 存在缺口 ⚠️

| ID | 标准 | 可测试性 | 备注 |
|----|------|----------|------|
| S-1 | 单列 > 50 任务时滚动 60fps | ✅ 可测试（但条件不严谨） | 60fps 依赖硬件基线，需明确测试设备规格（如 MacBook Pro M3 / 指定浏览器版本），否则 CI 环境可能不稳定 |
| S-2 | WebSocket 推送，更新延迟 < 1s | ⚠️ 缺少功能规格 | S-2 要求 WebSocket，但正文 M1-M3 中没有定义实时推送机制，属于验收标准与功能设计不一致 |
| S-3 | 点击复制后内容正确写入剪贴板 | ✅ 可测试 | Clipboard API 验证，但自动化测试中浏览器安全策略需要特殊处理（需 HTTPS 或 localhost） |
| S-4 | 多条件筛选结果准确 | ⚠️ 条件模糊 | "多条件"指哪些？AND/OR 逻辑？筛选器 UI 在正文中仅有 US-4 简单提及，无详细设计文档 |

### Could（可以有）- 描述不足 ⚠️

| ID | 标准 | 可测试性 | 备注 |
|----|------|----------|------|
| C-1 | 拖拽后状态机验证，非法流转拦截 | ⚠️ 规则未定义 | 哪些是"非法流转"？Kanban 拖拽是否允许双向（如 从 COL_4 拖回 COL_3）？若无明确规则，测试无法判定 Pass/Fail |

### 小结

**M-4、M-6、S-2、S-4、C-1 共 5 项需要补充定义**，否则无法形成有效的自动化测试判定条件。

### 2.2 关键边界条件清单（测试需覆盖但目前无定义）

1. **重试边界**：第 3 次重试失败后，任务状态是保持 `failed` 还是另有处理？
2. **并发冲突**：用户点击"取消"的同时后端 Agent 完成代码生成——以哪个为准？
3. **超长任务**：任务在 `running` 或 `testing` 状态超过 X 分钟无响应，是否有超时机制？超时后状态如何变更？
4. **网络分区**：取消请求发出但服务端超时，前端如何展示？
5. **Agent 失联**：执行中的 Agent 突然下线，任务是否自动进入 `failed`？
6. **cancel 按钮在 failed 状态是否显示**：表格中说取消按钮条件是 pending/running/testing，但 failed 状态也有取消路径（`failed → User_cancel → cancelled`），按钮是否该在 failed 态也出现？

## 三、功能边界清晰度评审

### 3.1 明确性问题

**Q1：Kanban 5 列 vs 7 个状态，边界不匹配**
- Kanban 列：待处理(COL_1) / 处理中(COL_2) / 测试中(COL_3) / 已完成(COL_4) / 已失败(COL_5)
- 状态机状态：pending / running / testing / passed / failed / cancelled / completed（7个）
- **缺口**：
  - `cancelled` 状态没有对应的 Kanban 列——取消后的任务消失了？还是有单独的"已取消"列？
  - `passed` 和 `completed` 两个状态都归入 COL_4 "已完成"，但两者是否在列内区分（颜色、标签）？用户体验上是否有感知？

**Q2：详情抽屉操作按钮缺失状态**
- 取消按钮：`pending / running / testing` 可见（正常）
- 重试按钮：`failed` 且重试次数 < 3 可见
- 查看代码按钮：`passed / completed` 可见
- **缺失**：当任务处于 `passed` 状态时，UI 上是自动进入 `completed` 还是要用户手动触发 Deliver 事件？用户看到 passed 态时应该做什么操作？

**Q3：passed → completed 的 Deliver 事件由 Agent_Deployer 自动触发**
- 如果是全自动，用户在 `passed` 态停留多久？
- 如果用户想手动确认后才交付，UX 上如何干预？

**Q4：筛选器功能无详细设计**
- US-4 提到筛选器，但没有任何筛选维度、筛选逻辑、UI 样式定义
- S-4 验收标准要求"多条件筛选结果准确"，但功能本身是黑盒

**Q5：重试机制的 retryCount 字段无人维护**
- 卡片字段定义了 `retryCount: 0`，流转规则表中未说明谁负责 +1
- 3 次上限由谁校验？前端还是后端？校验失败后的提示是什么？

## 四、状态机完整性评审

### 4.1 状态机图与流转规则表对比

| 问题 | 描述 | 严重程度 |
|------|------|----------|
| 流转规则表缺少 `retry` 事件 | 重试机制是核心功能，但 `failed → User_retry → running` 这条路径在流转规则表中完全缺失 | 🔴 高 |
| `running → cancelled` 路径缺失 | 规则表只写了 `pending/running/testing → cancelled`，没有明确的触发事件（是 User_cancel 吗？Agent 超时？系统中断？）| 🟡 中 |
| `passed → completed` 在图中未体现 | 流转图没有 `passed → completed` 这条边，但流转规则表和 M3.3 操作表都提到这条路径 | 🟡 中 |
| `testing → passed` 触发事件未定义 | 流转规则表写的是 `All_tests_pass`（类似事件名），但没有说明这是 Agent 自动触发还是需要外部确认 | 🟡 中 |
| `running/testing` 超时处理缺失 | 没有超时时间定义，也没有超时后的状态迁移（running 超时 → failed？还是挂起？） | 🟡 中 |
| `cancelled` 和 `completed` 是否有出度？ | 如果是终态，状态机图应明确标注为终态（双圆圈），目前图中未标注 | 🟡 中 |
| `failed(永久)` vs `failed` 是否是同一状态？ | 规则表提到 "failed(永久)" 但状态列表中只有 `failed` 一个状态，两者是同一状态的不同子类型还是两个不同状态？ | 🟡 中 |

### 4.2 状态机状态/路径汇总

```
状态（7个）：pending, running, testing, passed, failed, cancelled, completed

图中已有的流转（12条）：
  pending → running
  running → testing
  running → failed
  running → cancelled(?) ← 未在图中出现但规则表有
  testing → passed
  testing → failed
  failed → running (retry, <3次)
  failed → failed(永久) (= failed 终态)
  failed → cancelled
  passed → completed
  [pending/testing → cancelled]

缺失的流转（4+条）：
  ❌ running → cancelled (User_cancel 场景)
  ❌ pending → failed (超时/系统错误)
  ❌ running/testing → failed (超时场景)
  ❌ retry 事件的触发和计数逻辑
```

### 4.3 状态机漏洞总结

1. **`failed` 永久终态未建模**：重试 3 次用尽后进入永久 `failed`，但状态机图中未标注为终态，缺少出度
2. **`cancelled` 终态未标注**：作为终态，`cancelled` 不应再有后续流转，但图/表均未说明
3. **`running → cancelled` 用户取消路径缺失**：用户能否在 running 态取消？规则表只写"pending/running/testing → cancelled"但 running 的具体触发事件未定义
4. **状态到 Kanban 列的映射规则缺失**：状态机状态与 Kanban 列的映射关系未明确定义（`passed` 到底在 COL_4 还是 COL_3？`cancelled` 有列吗？）
