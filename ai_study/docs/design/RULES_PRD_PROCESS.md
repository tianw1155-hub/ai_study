# PRD 评审流程规范

> 状态：已生效 | 生效日期：2026-03-24

---

## 目的

确保每份 PRD 经过充分评审再进入开发，避免方向性错误。

---

## 评审流程

```
产品经理完成 PRD
        ↓
开发工程师 + 测试工程师 同时评审
        ↓
小毛球综合评审 + 终审
        ↓
小薇妹最终审阅
        ↓
通过后进入 Sprint
```

---

## 各角色职责

### 产品经理
- 撰写 PRD，确保包含完整的 MoSCoW、验收标准、技术方案
- 根据评审意见修改 PRD

### 开发工程师
- 评审维度：技术可行性、架构设计、开发成本、风险点
- 评审结果写入：`docs/prd/PRD-REVIEW-DEV.md`
- 结论：✅ 通过 / ⚠️ 需要修改 / ❌ 拒绝

### 测试工程师
- 评审维度：测试覆盖可行性、质量门禁设计、验收标准、风险点
- 评审结果写入：`docs/prd/PRD-REVIEW-TEST.md`
- 结论：✅ 通过 / ⚠️ 需要修改 / ❌ 拒绝

### 小毛球（总指挥）
- 汇总两份评审，出综合报告
- 裁定争议项
- 向小薇妹汇报

### 小薇妹（老板）
- 最终审阅
- 拍板进入 Sprint

---

## 评审触发条件

| 条件 | 是否触发评审 |
|------|-------------|
| 全新 PRD | ✅ 必须评审 |
| 已有 PRD 大改（超过 30% 内容变更） | ✅ 必须评审 |
| 已有 PRD 小改（修复错别字、格式） | ❌ 不需要 |
| Sprint 内紧急补充 | ⚠️ 可简化评审（30分钟快评） |

---

## 修订方案确认流程

当 PRD 需要修改时：
1. 小毛球先出修订方案
2. 小薇妹确认修订内容
3. 产品经理按修订意见修改
4. 修改完成后重新走评审流程

---

## 关键节点通知规则

以下节点必须直接弹给小薇妹：
- PRD 评审通过，可以进入 Sprint
- Sprint 内出现重大阻塞（超过 24h）
- 部署上线前需要确认
- 项目遇到风险需要升级
